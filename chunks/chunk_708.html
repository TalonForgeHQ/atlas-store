The five evidence questions a Windsurf procurement review must answer

1. Can every Cascade step be reconstructed from a per-step audit record?

A cascade_step_receipt should join cascade-step ID, tenant, developer identity, IDE install version, the chosen tool call (read_file / write_file / edit_file / run_terminal_command / search_code / browse_url), tool input hash, tool output hash, LLM-sub-processor version chosen by the multi-LLM router, temperature pin, system prompt version, context-window attribution, token-cost attribution, policy version in force at the step, human-approval bit, and timestamp. Without that join, “Cascade is auditable” stays a product claim instead of becoming evidence an enterprise buyer can test.

2. Can the multi-LLM router decision be exported per Cascade step?

The router currently exposes six sub-processors. Procurement needs per-step evidence that the router’s model-selection rule, the chosen sub-processor, the fallback chain, any cost-cap that fired, the region-of-processing used, the tenant-isolation token, and the per-step training-data-opt-out acknowledgement are all recorded as a router_decision_record. This join is the surface Fortune 500 + AmLaw 200 + EU-Commission-supplier audit teams will demand before any code touches Cascade.

3. Can every Windsurf Plugin marketplace grant be exported as evidence?

A plugin_grant_audit should pin the plugin manifest version, the plugin’s source license, the requested permission grant (filesystem / network / shell / clipboard / IDE API), the data-egress attestation, the install/rollback receipt, the per-IDE install checksum, and the per-customer allowlist enforcement. Windsurf’s marketplace model is unique among AI-coding vendors and is precisely the surface many enterprise buyers will want to scope down first.

4. Can every deployment mode prove its boundary?

Cloud multi-tenant, Enterprise-Cloud-Dedicated, and Self-Hosted Enterprise each have different evidence obligations. The deployment_boundary_receipt should pin mode + region + customer-controlled network boundary + KMS key id (or approved-key reference) + IDE install checksum + plugin allowlist per tenant + administrator approval + deploy timestamp. Self-Hosted Enterprise is the procurement-grade lever the cloud-only AI-coding vendors cannot reach.

5. Can per-tenant customer code, training, and retention be exported as a cascade?

A customer_data_lifecycle join should link tenant code isolation token, prompt retention policy version, training-data opt-out, context-attribution cascade, Enterprise-Cloud-Dedicated region pinning, Enterprise-Edu isolation, and per-IDE-install-per-developer. This is the per-customer surface GDPR Art. 28, CCPA/CPRA, EU AI Act Art. 14(4), and SOC 2 CC7.2 procurement will demand.

How to read this if you are a procurement, security, or AI-governance lead

  • Start at Cascade-step provenance — if you cannot export a single cascade_step_receipt, no downstream join will hold up under audit.
  • Treat the multi-LLM router as a sub-processor list — six vendors today, more likely tomorrow. The router_decision_record is what makes a future router expansion a safe one.
  • Scope the Plugin marketplace first — most Cascade-enabled tenants will default to a small allowlist. Get that allowlist exported as plugin_grant_audit evidence.
  • Pin deployment mode and region per tenant — Enterprise-Cloud-Dedicated and Self-Hosted Enterprise are the modes that survive the toughest audit. Cloud multi-tenant is fine for the long tail but should never be the path for regulated workloads.
  • Treat customer-data-lifecycle as a single cascade — tenant code isolation, prompt retention, training opt-out, context attribution, and region pinning all need to land on one per-tenant customer_data_lifecycle record.
