Why Letta is different from a memory API

Letta describes itself as infrastructure for stateful agents that remember, learn continuously, and improve over time. Its MemGPT lineage treats the context window as a constrained operating-system resource. Agents can rewrite self-editing core-memory blocks, retrieve archival knowledge, search recall history, and improve memory asynchronously through sleep-time agents. That makes every memory mutation part of the agent's control plane.

The resulting evidence model must connect memory formation, memory retrieval, context assembly, and downstream action without losing the source turn.

Self-editing memory needs source lineage

A block update is not just a database write. It changes the future prompt. The audit row must preserve the prior block version, triggering source message, edit operation, resulting block version, context-window version, and first downstream action. Without that join, a buyer cannot explain why the agent remembered an instruction—or prove that it removed one.

Core, archival, and recall memory have different deletion paths

Core memory is pinned context, archival memory is processed knowledge in external storage, and recall memory preserves interaction history. A GDPR deletion request can touch all three plus embeddings, caches, context-repository commits, and generated summaries. The evidence-gap map identifies native deletion evidence, custom hooks, and irreversible gaps for each store.

Sleep-time agents create asynchronous audit obligations

When memory improvement happens after the user-facing turn, the replay must preserve the sleep-time run, source evidence, context-repository commit, human override, rollback commit, and next production context that consumed the change. Otherwise the system can improve—or drift—between interactions without an accountable decision trail.
