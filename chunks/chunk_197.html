1. The 36 Veryfi-specific audit-gaps (grouped by evidence surface)

1.1 Per-mobile-device-fingerprint + per-mobile-os-version + per-mobile-app-version + per-mobile-sdk-version mobile-attestation-trail (5 audit-gaps)

  1. Per-mobile-device-fingerprint + per-mobile-os-version (iOS 17.x / Android 14.x) + per-mobile-os-build-number — every receipt/invoice OCR payload generated from the Veryfi White-Label Mobile SDK must be joined to the unique mobile-device-fingerprint + mobile-OS-version + mobile-OS-build-number at the time of capture. The audit trail needs the device-fingerprint + OS-version + OS-build + the SDK-version + the app-version + the device-region + the device-timezone at OCR-capture time.
  2. Per-mobile-app-version + per-mobile-app-bundle-id + per-mobile-app-installation-id — every OCR payload must trace back to the specific mobile-app-version + mobile-app-bundle-id (com.square.cash, com.intuit.quickbooks, com.veryfi.whitelabel) + per-installation-id. The audit trail needs the app-version + bundle-id + installation-id for the SOC 2 CC7.2 + EU AI Act Art. 12 + PCI-DSS 10.2.1.4 device-attribution evidence.
  3. Per-mobile-sdk-version + per-mobile-sdk-build-hash + per-mobile-sdk-deployment-channel (production/staging/canary) — the Veryfi SDK is the OCR-extraction-edge where most compliance breaches happen (mobile-SDK bypasses the server-side policy gates). The audit trail needs SDK-version + SDK-build-hash + SDK-deployment-channel + the SDK-init-timestamp + the SDK-API-key-id + the SDK-tenant-id.
  4. Per-mobile-network-type (wifi/cellular/ethernet) + per-mobile-carrier + per-mobile-carrier-country — the network-context matters for (a) IRS 1099-K + state-sales-tax jurisdiction-mapping (different states tax different line-items at different rates) + (b) GDPR + Schrems II data-residency routing (EU-customer-data must not transit non-EU carrier networks). The audit trail needs the network-type + carrier + carrier-country + the routing-decision made.
  5. Per-mobile-camera-permission-timestamp + per-mobile-photo-library-permission-timestamp + per-mobile-camera-roll-skip-event — the receipt/invoice photo capture path uses the mobile-camera + the mobile-photo-library. The audit trail needs the camera-permission-granted-timestamp + photo-library-permission-granted-timestamp + whether the photo was captured directly via SDK vs imported from the camera-roll (camera-roll import is a higher-risk path for receipt-tampering + receipt-replay attacks).

1.2 Per-OCR-token-id + per-OCR-confidence-score + per-line-item-bounding-box OCR-extraction-lineage (6 audit-gaps)

  1. Per-OCR-token-id + per-OCR-token-text + per-OCR-token-bounding-box + per-OCR-token-confidence-score (0.00-1.00) — every OCR-extracted token (merchant-name-token, line-item-description-token, tax-amount-token, currency-symbol-token, payment-method-token) must have a unique token-id + the token-text + the bounding-box (x,y,w,h in pixels) + the confidence-score. The audit trail needs the per-token attribution so the auditor can confirm "this tax-amount was extracted at confidence 0.94 from this bounding-box, and was verified by this human-reviewer-id at this timestamp."
  2. Per-line-item-id + per-line-item-bounding-box + per-line-item-text + per-line-item-merchant-name + per-line-item-merchant-category (grocery/dining/travel/software/hardware) — the line-item decomposition is the part that breaks during IRS-1099-K + state-sales-tax audits (a misclassified line-item-merchant-category produces the wrong tax-rate). The audit trail needs line-item-id + line-item-bounding-box + line-item-text + line-item-merchant-name + line-item-merchant-category + the merchant-category-confidence-score.
  3. Per-line-item-tax-amount + per-line-item-tax-rate + per-line-item-tax-jurisdiction — every line-item needs a tax-amount + a tax-rate + a tax-jurisdiction (CA-NY-TX-WA-EU-UK etc). The audit trail needs the tax-amount-precision (rounded vs exact) + the tax-rate-source (vendor-provided vs LLM-inferred vs merchant-disclosed) + the tax-jurisdiction-confidence-score. State-sales-tax + VAT + GST audits fail on this granularity.
  4. Per-line-item-currency + per-line-item-currency-conversion-rate + per-line-item-currency-conversion-timestamp — international receipts (EUR receipt on a US-business trip, GBP receipt on a EU-business trip) need currency + conversion-rate + conversion-timestamp + the FX-rate-source. The audit trail needs the currency + conversion-rate + conversion-timestamp + the FX-provider-id + the FX-provider-rate-version.
  5. Per-line-item-payment-method (cash/credit-card/debit-card/ACH/wire/crypto) + per-line-item-payment-method-last-four + per-line-item-payment-method-issuer — the payment-method attribution is critical for (a) IRS-1099-K reporting thresholds (different thresholds per payment-method) + (b) PCI-DSS scope-determination (a credit-card payment-method brings the receipt into PCI-DSS scope). The audit trail needs payment-method + last-four + issuer + issuer-country.
  6. Per-OCR-page-id + per-page-image-resolution + per-page-image-format + per-page-image-hash — every OCR-processed receipt/invoice page must have a page-id + the image-resolution (e.g. 1200x1600 px) + the image-format (JPEG/PNG/HEIC/PDF) + the image-content-hash (SHA-256). The audit trail needs the page-image + the page-image-hash to verify the OCR-processed image has not been retroactively swapped.

1.3 Per-LLM-model-id + per-LLM-prompt-hash + per-field-extraction-id + per-field-validation-id LLM-augmented-extraction-lineage (5 audit-gaps)

  1. Per-LLM-model-id (claude-sonnet-4-5 / gpt-4o / veryfi-custom-fine-tuned) + per-LLM-model-version-id + per-LLM-model-fine-tuning-run-id — when the field-level extraction is augmented by an LLM (vs pure OCR), the audit trail needs the LLM-model-id + model-version + the fine-tuning-run-id (for veryfi-custom models). EU AI Act Art. 9 (risk management) + Art. 11 (technical documentation) + Art. 12 (logging) require this granularity for any AI-system with material financial-output consequences.
  2. Per-LLM-prompt-hash + per-LLM-prompt-template-id + per-LLM-prompt-template-version-id + per-LLM-prompt-temperature + per-LLM-prompt-max-tokens — every LLM-call needs a prompt-hash + the prompt-template-id + the prompt-template-version-id + the temperature + the max-tokens. The audit trail needs the prompt-hash to be cryptographically verifiable so the auditor can reproduce the exact LLM-call that produced the field-extraction.
  3. Per-field-extraction-id + per-field-name (merchant_name / tax_amount / line_item_total / payment_method) + per-field-extraction-value + per-field-extraction-confidence-score — every extracted field (per-receipt + per-invoice) needs a field-extraction-id + field-name + field-value + field-confidence-score. The audit trail needs the field-extraction-id to join to the line-item-id + the OCR-token-id + the LLM-prompt-hash + the human-reviewer-id.
  4. Per-field-validation-id + per-field-validation-rule-id (regex / lookup-table / LLM-validate / cross-field-consistency) + per-field-validation-pass-fail — every extracted field goes through one or more validation-rules. The audit trail needs the validation-id + the rule-id + the pass/fail + the failure-reason. This is the evidence that the field was checked against business-rules before being passed to AP/ERP/accounting.
  5. Per-field-correction-id + per-field-correction-type (auto-corrected / human-corrected / rule-corrected) + per-field-correction-reason + per-field-correction-original-value + per-field-correction-new-value — when a field is corrected (auto-corrected by Veryfi, human-corrected by reviewer, rule-corrected by validation), the audit trail needs the correction-id + correction-type + reason + the original-value + the new-value. This is critical for IRS-1099-K + state-sales-tax audits where the field-correction-history proves the originally-extracted value was wrong + the correction was made before AP-write.

1.4 Per-human-reviewer-id + per-review-decision + per-correction-id + per-rollback-id human-in-loop-lineage (5 audit-gaps)

Per-human-reviewer-id + per-human-reviewer-role (AP-clerk / AP-manager / controller / CFO / external-auditor) + per-human-reviewer-authentication-id — every human-reviewer-action must have a unique reviewer-id + the role + the authentication-id (SSO/SAML/OIDC). The audit trail needs the reviewer-id + role + auth-id + the session-id + the session-duration.
  2. Per-review-decision (approve / reject / modify / escalate / re-route) + per-review-decision-timestamp + per-review-decision-reason — every review-event needs the decision + timestamp + reason. The audit trail needs the decision + timestamp + reason + the field-extraction-ids affected + the LLM-model-id used to assist the review.
  3. Per-correction-id + per-correction-by-human-reviewer-id + per-correction-affected-field-id + per-correction-original-value + per-correction-new-value + per-correction-reason — when a human-reviewer corrects a field-extraction, the correction-id joins to the reviewer-id + the field-extraction-id + the original + new value + reason. The audit trail needs this for any field that was modified by human-review.
  4. Per-rollback-id + per-rollback-trigger (validation-failure / human-rejection / batch-replay / fraud-detection / state-AG-subpoena) + per-rollback-affected-write-id + per-rollback-reason + per-rollback-human-reviewer-id — when an extraction (or an entire receipt batch) is rolled back (post-AP-write), the audit trail needs the rollback-id + the trigger + the affected-write-id + reason + the reviewer-id who authorized the rollback.
  5. Per-resolution-quality-score (0-100) + per-resolution-quality-rubric-version + per-resolution-quality-human-rater-id — every resolution (final-state of a receipt/invoice through OCR + LLM + human-review + AP-write) needs a resolution-quality-score + the rubric-version + the rater-id. The audit trail needs the quality-score + rubric-version + rater-id to prove the resolution was reviewed against a documented rubric.

1.5 Per-AI-action-id + per-action-type + per-action-rollback-id write-action-lineage (5 audit-gaps)

  1. Per-AI-action-id + per-action-type (clause-insert / clause-replace / clause-delete / clause-rewrite / risk-flag-set / precedent-insert) — when the Spellbook agent makes a write into the draft (clause-insert, clause-replace, clause-delete, clause-rewrite, risk-flag-set, precedent-insert), the action-id needs to be unique + immutable + cross-system-joined (Spellbook → Word/Google Docs → DocuSign → iManage/NetDocuments → client-matter-management). The audit trail needs the action-type + the action-target-clause-id + the action-timestamp.
  2. Per-AI-action-override-id + per-override-reason + per-override-human-reviewer-id — when the supervising attorney + the partner + the GC overrides an AI action, the override-id + override-reason + override-human-reviewer-id must be in the same row as the action-id. ABA Model Rule 5.1 + 5.3 require this evidence to demonstrate the attorney exercised reasonable supervisory oversight of the nonlawyer assistance.
  3. Per-action-rollback-id + per-rollback-timestamp + per-rollback-trigger-event — when an AI-action is rolled back (post-write detection of hallucination, error, or compliance violation), the rollback-id + rollback-timestamp + rollback-trigger-event must be in the audit trail.   4. Per-action-rollback-reason + per-action-rollback-policy-id + per-action-rollback-citation-to-policy — the rollback-reason must reference the specific policy-id that triggered the rollback. Per-write-target-system-id (Word / Google Docs / DocuSign / iManage / NetDocuments / client-matter-management) + per-write-target-system-action-id + per-write-confirmation-timestamp — the AI-action must be traceable through to the target-system (Word/Google Docs/DocuSign/iManage/NetDocuments/client-matter-management). The audit trail needs the target-system-id + the target-system-action-id + the target-system-confirmation-timestamp.

The audit trail needs the signal-id + the rule-version + the action-taken (block / quarantine / alert / allow-with-warning).

  3. Per-OCR-time + per-LLM-call-time + per-human-review-time + per-AP-write-time (end-to-end latency attribution) — the end-to-end processing time must be attributable per-stage. The audit trail needs the OCR-time + LLM-call-time + human-review-time + AP-write-time + the total-time + the SLA-breach-event-flag.
  4. Per-prompt-template-version + per-prompt-template-approval-status + per-prompt-template-approval-human-reviewer-id — every prompt-template-version must have an approval-status + the human-reviewer-id who approved it. Per-document-type (receipts / invoices / W-2 / 1099 / bank statements / credit-cards / business-cards) + per-document-type-retention-policy + per-document-type-data-residency-policy — different document-types have different retention requirements (W-2 = 4 years per IRS, 1099 = 4 years, bank-statements = 7 years per BSA, credit-card = PCI-DSS scope). The audit trail needs the document-type + the retention-policy-id + the data-residency-policy-id per receipt/invoice.

  1. Per-document-id + per-document-type + per-document-upload-timestamp + per-document-upload-source (mobile-SDK / web-upload / email-forward / API-direct) — every document must have a unique document-id + document-type + upload-timestamp + upload-source. The audit trail needs the source for the SOC 2 CC6.1 + CC7.2 + EU AI Act Art. 12 device-attribution.
  2. Per-document-source-account-id + per-document-source-tenant-id + per-document-source-user-id — multi-tenant SaaS deployments must have per-tenant + per-user attribution. The audit trail needs the source-account-id + source-tenant-id + source-user-id + the user-role + the user-authentication-method.
  3. Per-document-completeness-score (0-100) + per-document-completeness-missing-field-list — every extracted document gets a completeness-score that reflects how many required fields were extracted. The audit trail needs the score + the missing-field-list for downstream AP-write-decision.
  4. Per-document-fraud-score (0-100) + per-document-fraud-detection-rule-id + per-document-fraud-detection-trigger — every receipt/invoice goes through fraud-detection (duplicate-detection, amount-anomaly, merchant-anomaly, photo-tampering-detection). The audit trail needs the fraud-score + the rule-id + the trigger.
  5. Per-document-lineage-completeness-score (0-100) + per-lineage-missing-step-list — a rollup of how complete the document-lineage is across all surfaces (mobile-device-fingerprint + OCR + LLM + human-review + AP-write). The audit trail needs the lineage-completeness-score + the missing-step-list for SOC 2 + EU AI Act + ISO 42001 readiness.

  1. Walk me through your per-mobile-device-fingerprint + per-mobile-sdk-version mobile-attestation-trail.   11. For state-sales-tax compliance, show me the per-line-item-tax-jurisdiction + per-line-item-tax-rate-source (vendor-provided vs LLM-inferred vs merchant-disclosed) + per-line-item-tax-amount-precision (rounded vs exact). (state-sales-tax + state-AG consumer-protection + IRS Pub 1075)
  12. For EU AI Act Annex III §4 essential-services operators (financial-services + insurance + healthcare), show me the per-LLM-model-version-id + per-LLM-model-fine-tuning-run-id + per-prompt-template-approval-status + per-prompt-template-approval-human-reviewer-id. (EU AI Act Art. 9 + 11 + 14 + 15 + 50 + 53(d) + ISO 42001 §6.1.2 + §9.4)
  13. For audit-trail completeness, show me the per-document-lineage-completeness-score + per-lineage-missing-step-list. What percentage of documents have a fully complete lineage vs partial-lineage?   14. For multi-tenant SaaS deployments, show me the per-document-source-account-id + per-document-source-tenant-id + per-document-source-user-id + per-user-role + per-user-authentication-method. 3. The 5-layer reference architecture (Veryfi-specific)

  1. Layer 1 — Mobile-capture + Mobile-SDK-edge: White-Label Mobile SDK captures receipt/invoice via device-camera + photo-library. SDK-version + OS-version + device-fingerprint + network-context + region-context all stamped at capture. SDK-version pinned + SDK-build-hash verifiable. Per-mobile-SDK-version lineage is the evidence primitive most mobile-IDP vendors skip.
  2. Layer 2 — OCR-extraction + Tokenization: OCR pipeline extracts per-token + per-bounding-box + per-confidence-score. OCR-token-id + OCR-token-text + OCR-token-bounding-box + OCR-token-confidence-score all exported. Line-item decomposition produces per-line-item-id + per-line-item-merchant-name + per-line-item-merchant-category + per-line-item-tax-amount + per-line-item-tax-rate + per-line-item-tax-jurisdiction + per-line-item-currency + per-line-item-payment-method.
  3. Layer 3 — LLM-augmented-extraction + Field-correction: For ambiguous fields (low OCR-confidence, missing-fields, cross-field-inconsistencies), an LLM is called with a prompt-template-version + a prompt-hash + the LLM-model-id. The LLM-output is verified against validation-rules. Per-field-extraction-id + per-field-validation-id + per-field-correction-id + per-LLM-model-id + per-LLM-prompt-hash all exported.
  4. Layer 4 — Human-in-loop review + Rollback: For material-value receipts (>$1000), high-fraud-score, low-confidence-extraction, regulatory-flag (W-2 / 1099 / bank-statement), a human-reviewer is required. Per-human-reviewer-id + per-review-decision + per-correction-id + per-rollback-id all exported. Reviewer-role + reviewer-authentication-id + reviewer-session-duration all exported.
  5. Layer 5 — AP-write + Audit-trail-completeness: Final extracted + reviewed + corrected fields are written to AP/ERP/accounting (NetSuite / QuickBooks / SAP / Coupa / Workday). Per-write-target-system-id + per-write-target-system-action-id + per-write-confirmation-timestamp all exported. Audit-trail-completeness-score + lineage-missing-step-list computed at this layer.
