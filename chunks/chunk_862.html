2. The 16-column provenance cascade

Every ai_security_compliance_attestation vendor that ships a Trust Management Platform must answer five audit questions in writing:

Q1 — Per-organization_id join-table reproducibility

Can you reproduce, on demand, the full join-table for any single customer tenant: organization_id → framework_id → framework_version_id → control_id → control_version_id → evidence_request_id → evidence_artifact_id → evidence_artifact_sha256 → evidence_artifact_collector_integration_id → evidence_artifact_collector_integration_version → auditor_id → audit_period_start → audit_period_end → control_test_result → integration_health_score → remediation_ticket_id?

If the vendor's customer-success team has to escalate to engineering to answer this, the answer is no. If the customer-success team can answer it from a single SQL query against an evidence-exported view, the answer is yes. The gap between those two answers is what we map in the $500 / 48h evidence-gap-map deliverable.

Q2 — Per-evidence-artifact-sha256 immutable hash trail

Evidence collection without an immutable artifact hash is not audit-grade. The auditor must verify that the evidence presented today is byte-for-byte identical to the evidence presented yesterday. The receipt needs:

  • per-evidence_artifact_id: the immutable artifact identifier (UUID v4 or v7)
  • per-evidence_artifact_sha256: the cryptographic hash that pins the byte-content at the moment of collection
  • per-evidence_artifact_collector_integration_id: which integration (AWS Config + GCP Cloud Asset Inventory + Azure Policy + GitHub + Okta + Jamf + Vanta API + ...) collected the artifact
  • per-evidence_artifact_collector_integration_version: which version of that integration ran (so a re-collect against a newer version does not invalidate the audit trail)

Q3 — Per-framework-version + per-control-version governance

Frameworks evolve. SOC 2 2017 TSC is not SOC 2 2022 TSC. ISO 27001:2013 is not ISO 27001:2022. ISO 42001 did not exist before 2023. PCI DSS 3.2.1 is not PCI DSS 4.0. The receipt needs framework_version_id + control_version_id + control_test_result per audit_period so the auditor knows which version of which control was tested in which period.

Q4 — Per-integration-health-score liveness

An auditor needs to know not just what evidence was collected, but whether the integration is still alive. A stale integration that has not reported in 30 days is a different control-state than a live integration reporting every 15 minutes. integration_health_score + evidence_artifact_collector_integration_version + per-evidence-freshness timestamp together prove the control is operating today.

Q5 — Per-auditor-portal sub-processor-cascade

The auditor (Big-4 or boutique) needs a portal where they can pull evidence on demand. The receipt needs auditor_id + audit_period_start + audit_period_end + per-auditor-portal-access-log so the auditor's own audit-log can prove who pulled what when, and so a SOC 2 + ISO 42001 audit-period boundary is unambiguous.
