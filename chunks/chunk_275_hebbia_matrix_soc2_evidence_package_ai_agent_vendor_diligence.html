1 · Why a vanilla SOC 2 letter is not enough for an AI agent vendor

A "SOC 2 Type II" letter is a one-page attestation that, for the period in scope, the controls the vendor claimed to have were operating effectively. For a CRM, that letter tells you most of what you need: who can log in, where the data lives, how access is reviewed, how incidents are triaged, how change is approved.

For an AI agent vendor — Hebbia, Glean, Harvey, Cohere, Moveworks, Spellbook, an internal copilot builder, a custom RAG stack on top of OpenAI, Anthropic, Cohere, or Mistral — the same letter barely moves the needle. The auditor's tests did not exercise retrieval, did not exercise tool calls, did not exercise model invocation with customer data. The critical control points are invisible in a Type II report because the auditor and the vendor agreed the controls were about human-operated infrastructure: identity, change, availability, confidentiality, processing integrity — all in the classical sense.

What you actually need is a parallel evidence package that lays down, on paper, how the vendor's AI-plane behaves in five places classical SOC 2 can't see: prompt inputs, retrievals, model outputs, tool invocations, and human-oversight decisions. That package is what we mean by SOC 2 evidence package for AI agent vendor diligence, and it is exactly the gap you, the buyer, have to close before signing a contract with Hebbia, Glean, Harvey, Cohere, or any other AI agent vendor.

Bottom line: if your vendor's response to a TPRM questionnaire cites only the SOC 2 letter, they have not answered the question. Use this article as the question.

2 · The five TSC criteria that break on agentic systems

The 2017 Trust Service Criteria have ten principles in CC1-CC9. Five of them are where AI agents genuinely change the risk profile, and each one needs an additional evidence row the auditor never tested:

TSC Plain meaning Why AI agents break it
CC6.1 Logical access to production systems A RAG retriever is a production system. Per-tenant namespace, per-document ACL reflection, per-citation authorization have to be enforced at retrieval time, not at original-ingest time.
CC6.7 Restriction of data flow to authorized users Prompt and context cross-tenant leakage is a CC6.7 failure surface. Embeddings plus retrievals plus model invocations all have to be partition-respecting.
CC7.2 System monitoring and intrusion detection An agent that calls 14 tools to complete one user request is a 14-statement side-effect chain that has to be monitored, not just network traffic.
CC7.3 Incident detection and response Prompt injection that exfiltrates data, jailbreaks that hit the safety boundary, retrievals that surface another tenant's data — each is an incident and needs a runbook entry.
CC9.2 Vendor and business partner risk The sub-processor list for an AI vendor is longer than for a SaaS, because every model API and every retrieval vector store is itself a sub-processor the customer has to approve.

Each of these five rows needs an evidence document you can drop into your TPRM binder. Sections 3-7 below describe what those documents should contain.

4 · RAG retrieval: CC6.1 controls that actually matter

The classical SOC 2 controls tested who could log in to the production environment. For a RAG system, you also need to ask: who is the user asking, which documents are retrievable to that user, and how is that authorization reflected at retrieval time?

Evidence patterns to demand:

  • Per-tenant namespace: every embedding, every document chunk, every metadata field is keyed to a tenant_id.
  • Per-document ACL reflection: the retriever filters on the union of (tenant_id, document_id ACL, user role) before scoring.
  • Re-ranking audit: confidence score per retrieved chunk; chunks below threshold are dropped and the drop reason is logged.
  • Cross-tenant leakage testing: a quarterly red-team that asks "show me another customer's Q3 contract" and verifies the system refuses + logs the attempt.
  • Embedding redaction: any embedded text containing PII is redacted before the embedding call and the redaction log is preserved.

Without these, the vendor can pass classical SOC 2 but still produce a CC6.1 violation the day a customer's user runs the prompt that retrieves another customer's document.

5 · Agent tool execution: CC7.2 in practice

An AI agent that can call tools — email senders, file uploaders, ticket creators, payment processors, calendar mutators — has a CC7.2 problem the auditor never tested: a single user request can trigger dozens of side effects. The monitoring that detects intrusion on a web app does not detect a malicious prompt that walks the agent through 14 tools to exfiltrate data.

Evidence patterns to demand:

  • Tool-call allowlist: only pre-approved tools are invokable by the agent; no arbitrary shell or arbitrary HTTP.
  • Side-effect declaration: each tool declares its side-effects (read, write, mutate external system); the agent must explicitly declare which side-effects the response will trigger before invocation.
  • Per-call authorization: high-side-effect tools (write, send, post, pay) require either user-role membership or human-in-the-loop approval.
  • Rate-limit and circuit-breaker: each tool has a per-tenant rate limit and a circuit breaker that opens after N failures.
  • Side-effect log: every tool call is logged with input, output, side-effect class, agent reason, prior-step retrievals (for replay), and user identity.

This is the difference between an agent that could be granted SC-2 in the classical sense and an agent that actually meets CC7.2 in the modern sense.

6 · Human-in-the-loop: mapping CC9.2 to approval logs

EU AI Act Article 14 demands "human oversight" for high-risk systems — and a SOC 2 Type II report has nothing to say about that. The way to bridge is to extend the CC9.2 vendor-risk evidence with a human-in-the-loop evidence row:

  • Approval threshold policy: which outputs require human approval before they are returned to the user or written to the system of record.
  • Approval routing: which user(s) the approval goes to, in what order, with what escalation.
  • Approval interface: where the human sees the agent's draft, what they can edit, what they can reject, what they can defer.
  • Approval logging: every approval, edit, rejection, deferral is logged with timestamp, agent session id, agent reason, human identity, and decision latency.
  • Approval analytics: median + 95p decision latency, override rate per agent type, repeat-approval rate (operator rubber-stamping detection).

This is the kind of evidence a buyer can hand to a regulator, an auditor, or a board member and answer the question, "did a human approve each action that mattered?"

7 · Deletion evidence for GDPR Art. 17 cascade

When a customer invokes Art. 17 (right to erasure) the classical SaaS vendor says "we ran a SQL DELETE on your row, here is the receipt." For an AI vendor, the same invocation has to propagate through every system that touched the data:

  1. Source-system row deletion in the original product database.
  2. Search-index deletion (Elastic / OpenSearch / typesense).
  3. Vector-store deletion (Pinecone / Weaviate / Qdrant / pgvector).
  4. Embedding-cache invalidation or scheduled flush.
  5. Model-cache invalidation (any KV-cache replay store or in-flight prompt completion).
  6. Audit-log redaction (raw content + metadata; not just the row id).
  7. Backup-cycle gap: until backups age out and the deletion propagates to cold storage.
  8. Sub-processor notification: model provider, embeddings provider, cloud region, etc.

The evidence to demand is a runbook that walks every step plus a sample receipt showing a real Art. 17 invocation completed end-to-end inside the contractual SLA (30 days for GDPR, 45 days for CCPA §1798.105 with extension).

10 · FAQ — what CISOs ask in week one

Q1. The vendor sent a SOC 2 Type II letter last week. Do I still need this package?

Yes. The letter covers classical controls. The 12 AI-plane documents above are not in the Type II scope for any of the major vendors we have audited.

Q2. My procurement contract says "SOC 2 Type II compliant." That should be enough, right?

No. That clause only says the vendor has a SOC 2 letter; it doesn't say the AI plane is in scope. You need a contract addendum or an evidence package that names the AI plane explicitly.

Q3. The vendor claims ISO 42001. Same answer as SOC 2?

ISO 42001 is the AI-management-system certification (2023). It is closer to what you want but still does not produce the AI-plane evidence bundle by itself. Demand the binder, then map each binder document to the ISO 42001 controls (A.5-A.18) and you have a tighter due-diligence file.

Q4. What about EU AI Act Article 14?

Article 14 is the operational human-oversight requirement for high-risk AI. The human-in-the-loop approval logs in document 9 of the binder are the closest first-party evidence the vendor can produce. Pair it with Article 12 (log retention) and Article 50 (transparency labeling) for a complete Art. 14 file.
