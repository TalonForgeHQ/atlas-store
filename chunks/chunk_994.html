Veza Access Graph vs C1 Universal Identity Graph: AI Agent Identity Governance Evidence Surfaces

Published July 23, 2026 · Atlas Audit Prep · long-tail keyword: Veza Access Graph vs C1 universal identity graph

The ai_agent_identity_governance cohort has two distinct identity-plane products: Veza, which publishes a named Access Graph that connects identities to data; and C1 (formerly ConductorOne), which publishes a named universal identity graph that connects users, services, agents, and API keys. The two planes answer overlapping but non-identical buyer questions.

Research basis: first-party product pages for Veza (veza.com homepage, /product, /company) and C1 (c1.ai /about-us, /platform) were verified on 2026-07-23. This is an evidence-surface framework, not a claim that either vendor fails a control.

Where the two planes overlap

Both vendors publish a graph product. Both publish an AI-agent-security product surface. Both publish an NHI (non-human identity) governance surface. Both integrate with Okta, AWS, and the major SaaS identity providers. Both target identity-governance and security teams, and both aim to be evidence-exportable for SOC 2 Type II and ISO 27001 audits.

Where the two planes differ

  1. Graph subject: C1's universal identity graph is subject-centric (every user, service, agent, and API key is a node). Veza's Access Graph is authorization-centric (every subject → permission → resource → action → data_record edge is a first-class node).
  2. Agent-action product surface: C1 publishes an Identity-Aware Gateway + C1 Autonomous Worker + Headless Identity Infrastructure stack for AI-agent request paths. Veza publishes a separately-named Access Agents + Access AI + Access Hub product surface for the same job, with the agent-to-data authorization edge as the primary node.
  3. Governance flow plane: C1 emphasizes lifecycle (registration, approval, review, JIT, revocation) plus policy-as-code guardrails. Veza emphasizes Access Reviews + Lifecycle Management + Access Requests + Access AuthZ + Separation of Duties, with the Access Graph as the substrate that all five flows read from.
  4. Integration breadth vs depth: Veza publishes 325+ integrations with named top-14 (AWS, Crowdstrike, Google Cloud, Microsoft Azure, Open Authorization API, Okta, Snowflake, Active Directory, Databricks, GitHub, Google Drive, OpenAI, Oracle, Salesforce). C1 emphasizes the identity-graph + MCP-aware governance plane, with named customers Ramp + DigitalOcean + Brex + Instacart + Qualtrics + Zscaler + PriceSmart + DailyPay.
  5. Origin story wedge: C1's two co-founders (Alex Bovee and Paul Querna) met on Okta security products per the first-party Founded-in-Security copy. Veza's three co-founders (Tarun Thakur, Maohua Lu, Rob Whitcher) started the company in early 2020 with a cloud-data-authorization origin story — "the world's data was rapidly, irreversibly transitioning to the cloud" — per the first-party /company page.

The 22-surface evidence-gap map

For a buyer comparing the two on a single agent-identity-governance procurement, the audit fields that matter are:

  1. access_graph_node_id — stable Veza node identifier per (subject, permission, resource, action, data_record).
  2. access_graph_edge_id — stable Veza edge identifier per (subject → permission → resource).
  3. subject_identity_id + permission_id + resource_id + action_id + data_record_id — five-field join.
  4. tenant_id + cross_tenant_no_bleed_audit_trail — proves one tenant's evidence is not retrievable from another tenant's export.
  5. edge_freshness_timestamp — last time the Access Graph edge was re-evaluated, not last time it was queried.
  6. credential_rotation_event_id — every rotation produces a retrievable event with prior + new credential hash.
  7. agent_identity_id + tool_call_id + access_decision — per-agent-action trail with allow/deny + reason.
  8. veza_policy_id + veza_policy_version — per-decision evidence-pinning.
  9. downstream_resource_action — what happened to the resource after the allow decision.
  10. revocation_event_id + change_request_id + audit_export_id — the join between Access Graph state change and the audit-log export that proves it.
  11. downstream_system_state_change — proof the revocation propagated.
  12. integration_id + per-integration-evidence-export-retrieval — per-integration (AWS, Okta, Snowflake, Databricks, GitHub, OpenAI, Salesforce) evidence.

Procurement questions

  • Can the Access Graph export prove a single (subject → permission → resource → action → data_record) join with a stable edge_id, not just a query result?
  • Are agent actions traceable from Access Agents through tool_call_id to the downstream resource change?
  • Does the integration coverage export 325+ integrations with per-integration per-tenant evidence, or only a count?
  • Are revoked edges still retrievable, with prior-state hash and rotation_event_id?
  • Can the same export prove cross-tenant-no-bleed plus per-policy-version evidence-pinning plus per-credential rotation?
