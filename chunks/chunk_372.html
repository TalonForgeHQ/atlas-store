The Penpot AI + MCP surface (verified live 2026-07-19)

Penpot exposes four distinct surfaces that an external AI agent can interact with:

  1. Penpot AI Workflows — penpot.app/ai/ai-workflows exposes AI-driven component generation, token-system mutation, and design-system-driven handoff. AI workflows can read a Penpot file's components + tokens + typography + color variables, then emit updated versions back to the same file.
  2. Penpot MCP Server — a Model Context Protocol bridge that lets external AI agents (Claude Code, Claude Desktop, Cursor, VS Code, Codex, and any other MCP-compatible client) read and mutate live Penpot files. The MCP server runs where the customer runs it — on Penpot SaaS, on the customer's self-host instance, or behind the customer's firewall.
  3. Plugin allowlist (Enterprise tier) — security teams pin exactly which plugins are permitted to run in a workspace. Untrusted plugin code is blocked at the workspace level, which is the only cohort-sibling plugin governance model in ai_design_tools.
  4. Integration API + Webhooks — penpot.app/integrations-api exposes the same file/board/page/component/token surface to external systems (Slack, Teams, Google Workspace, Jira, GitHub, custom MCP-bridged tools), with read/write capability and per-team allowlist.

The five audit gaps that EU regulated buyers will ask about

Gap 1 — AI Workflow → design-file mutation provenance. When an AI workflow mutates a Penpot file (changes a token, adds a component, regenerates a page), buyers need the full chain: AI-workflow-prompt → model version (which Claude / GPT / Gemini / open-weights model?) → retrieved design context (which file, which tokens, which components?) → plan-emitted tool calls → file/board/page/component/token/style mutation → version-history entry. Penpot's AI Workflows surface does not publish this chain by default.
Gap 2 — MCP-server → Penpot-file mutation provenance. When a Claude Code / Cursor / Codex session invokes an MCP tool that mutates a Penpot file, buyers need the same chain: MCP-client session ID → prompt → model version → retrieved Penpot file context → MCP tool call → file mutation → MCP-server-write log → version-history entry. The MCP server surface is new (2025-2026 cohort-wide), and Penpot's documentation does not yet publish a complete MCP-write audit trail.
Gap 3 — Plugin → Penpot-file mutation provenance. When an Enterprise-tier allowed plugin mutates a Penpot file (e.g., a custom internal plugin that auto-generates components from a Jira ticket), buyers need: plugin ID → plugin version → plugin-trigger context → plugin-emitted tool call → file mutation → plugin-write log. Plugin allowlist governs which plugins CAN run; it does not by itself produce a complete mutation audit trail.
Gap 4 — Integration-API → Penpot-file mutation provenance. When an external system (Slack, Teams, Jira, GitHub, custom MCP bridge) writes to a Penpot file via the Integration API, buyers need: API caller identity → API auth scope → payload contents → payload-to-file mutation mapping → API-write log → version-history entry. Per-team allowlist governs which integrations CAN write; it does not by itself produce a complete mutation audit trail.
Gap 5 — Self-host-instance isolation + cross-instance propagation. When a customer self-hosts Penpot, the data never leaves the customer's infrastructure. But buyers still need: which self-host-instance processed the AI workflow / MCP tool call / plugin write / Integration API write, which tenant (workspace / team / file) within the instance, and whether any cross-instance sync / backup / replication propagated the mutation. MPL-2.0 source-auditability is necessary but not sufficient — buyers also need instance-level evidence.
