Why durable execution still needs an audit receipt

Temporal preserves workflow state and event history so applications can recover after failures. Enterprise diligence asks a narrower evidence question: can one exported record explain which workflow ran, which code and worker handled it, which activities retried, which external messages changed state, and which operator intervened?

A workflow-history UI can help an engineer investigate one run, but a buyer-ready control needs stable identifiers, tenant boundaries, retention policy, and an integrity check. The objective is not to replace Temporal history. It is to turn the history and its surrounding control metadata into a replayable audit receipt.

Five controls for defensible workflow replay

  1. Pin run identity: never treat a mutable workflow ID as a substitute for the exact workflow run ID.
  2. Join workers to history: retain the task queue, worker build, and deployment version beside every relevant event.
  3. Explain retries: preserve activity attempt numbers and the retry-policy version that governed each decision.
  4. Attribute external change: distinguish schedules, signals, updates, cancellations, terminations, and operator actions.
  5. Sign the export: validate namespace and tenant scope, then hash the serialized replay receipt.

Audit questions the map answers

  • Can a business outcome resolve to one exact Temporal workflow run and event history?
  • Can each activity failure and retry be explained by the active retry policy?
  • Can a reviewer identify the worker build that processed each task?
  • Can operator intervention be separated from application-driven signals and updates?
  • Can an export prove tenant isolation and remain verifiable after retention expires?
