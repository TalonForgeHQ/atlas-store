Evidence replay drill

For each requested artifact, the buyer should be able to start with a vendor identifier and finish with a reproducible evidence chain. The minimum chain is: source-feed version → finding or quantification event → FocusTag class → mapped standard → control version → evidence hash → rating output → remediation recommendation. The chain should be scoped by tenant, portfolio, vendor, and model version so a shared platform cannot accidentally mix customer records.

For FAIR outputs, preserve the input assumptions and model version alongside the loss-frequency and loss-magnitude calculations. For FocusTags, preserve the taxonomy version and the finding IDs that caused each tag. For Standards Methodology, preserve the named standard, control revision, and citation anchor. For Ransomware Susceptibility, preserve the score update, confidence interval, and mapped technique.

Procurement questions for the Black Kite team

  • Which FAIR model version and data-source versions are attached to every Cyber Risk Quantification event?
  • Can a customer export a complete FocusTags history with tenant, vendor, finding, taxonomy, and confidence identifiers?
  • How does Standards-Based Data record control-version changes when NIST, ISO, SOC 2, PCI, or HIPAA mappings are revised?
  • Can Ransomware Susceptibility score changes be replayed from source observation through recommendation tier?
  • Does the platform provide an auditable bridge between a material cyber incident, a rating update, and the underlying quantification event?

These questions are deliberately evidence-oriented rather than claims about Black Kite’s internal controls. The first-party pages establish the product surfaces; the buyer’s procurement review must establish the depth, retention, exportability, and tenant-isolation behavior of each corresponding audit trail.
