The cross-plane lineage problem

Comet spans traditional machine-learning experiment and model lifecycle evidence while Opik covers LLM and AI-agent evaluation, tracing, prompt management, and production monitoring. An audit-ready implementation must join those planes. A production agent trace should resolve to the exact dataset version, training run, artifact, registered model version, prompt version, evaluator version, deployment, and human annotation that explain the behavior.

Keeping experiment evidence and agent observability in separate exports creates a replay gap. The gap appears when a reviewer asks which model artifact served a trace, which evaluation approved it, whether the prompt changed after approval, or whether two tenants could ever share an identifier.

Five controls that make the receipt defensible

  1. Immutable version joins: store dataset, model, prompt, and evaluator version identifiers rather than mutable names.
  2. Trace-to-model resolution: make every production trace resolve to one deployment and one registered model version.
  3. Promotion evidence: preserve the evaluation score and human annotation used to approve a deployment.
  4. Tenant-scoped keys: prefix and validate every workspace, project, trace, and artifact lookup with tenant_id.
  5. Replay export: generate one machine-readable bundle that reconstructs the experiment-to-agent decision chain without dashboard access.

Where Opik is distinct in the MLOps stack

Opik adds an open-source, self-hostable LLM and agent evaluation boundary to Comet's experiment, artifact, and model-registry surface. That makes deployment topology part of the evidence map: the same receipt should work for a managed deployment or a customer-controlled Opik environment. It also makes evaluator lineage and prompt-to-model joins first-class rather than leaving them in an unrelated observability silo.

Audit questions this map answers

  • Can an Opik trace identify the exact Comet model and dataset versions behind a response?
  • Can a reviewer reproduce the evaluator and score used before model promotion?
  • Can a prompt change be distinguished from a model change during incident review?
  • Can a human annotation be connected to the trace and deployment it approved?
  • Can the export prove that no cross-tenant identifier was accepted during replay?
