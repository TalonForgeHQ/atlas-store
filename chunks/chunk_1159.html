Why on-prem ML model registries break in regulated buyers

Regulated AI buyers — Tier-1 banks, hospital networks, defense contractors, public-sector statisticians — frequently cannot fully delegate model-lineage to a multi-tenant SaaS. Compute and data must remain on customer-controlled hardware, yet the registry, the experiment history, and the deployment approvals must still support a single replayable chain that a regulator can read. A registry that can only answer "what is the latest model version?" fails the moment a buyer asks "what was the exact configuration, environment, and data state that produced this deployed version, and who approved promotion?"

Valohai's first-party positioning — "The MLOps platform for AI products that combine LLMs and specialized models. Compare, evaluate, and ship configurations against your real data, on your own cloud, with full lineage from data to deployment" — targets exactly that buyer. Its execution substrate runs on customer AWS, Azure, Google Cloud, or Oracle Cloud tenants, and the registry and experiment records are preserved across every run.

Five controls for defensible on-prem replay

  1. Bind execution to cloud credential: every Valohai execution must terminate through a customer-controlled short-lived credential, so the audit trail can be signed by the buyer.
  2. Pin code to execution: preserve the customer repository commit and notebook commit alongside the execution identifier, so the precise revision is recoverable.
  3. Distinguish approval from deployment: a model version should not reach a serving target without a resolvable promotion record signed by named approvers.
  4. Record environment topology: identify whether evidence originated on customer on-prem HPC, an AWS or Azure tenant, a private VPC, or an air-gapped deployment.
  5. Export one replay bundle with cross-tenant proof: serialize the full chain, validate tenant scope, and sign it with replay_hash.

Audit questions the map answers

  • Can a deployed model resolve to the exact execution, environment, code revision, and dataset version?
  • Can each model version resolve to a separate evaluation metric run and approval record?
  • Can a regulator distinguish a customer-cloud adoption from a SaaS adoption deterministically?
  • Can the replay prove that no identifier crossed customer tenant boundaries?
  • Can the bundle be re-validated three years later without contacting Valohai?
