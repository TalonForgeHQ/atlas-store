Cascade 1 — Palmyra model release → model documentation + evaluation + safety-policy + training-data-summary

For every Palmyra model release used by a customer workflow, the evidence pack must export one immutable receipt joining:

  • Model ID + version — e.g. palmyra-x-002 + release_date + deprecation_policy + retention_window.
  • Model documentation version — the model card + system card + capability description pinned to the exact release the customer was provisioned against, not the marketing summary.
  • Evaluation version — benchmark scores + red-team results + jailbreak-resistance runs pinned to that model version, with the evaluation harness version + dataset version + date.
  • Safety-policy version — content-policy + refusal-policy + jailbreak-handling-policy pinned to the model version, with policy effective-date + owner + change-log.
  • Training-data-summary version — EU AI Act Article 53(1)(b) GPAI training-data-summary publication version + GDPR Art. 28 sub-processor training-data-opt-out flag.

This is the join that proves "you ran model X with policy Y against training data Z" without forcing procurement to chase four different trust pages and a status page to reconstruct it.

Cascade 2 — WRITER Agent run → activity trace + source citations + decision log + permissions + approvals + control versions

For every WRITER Agent run, the evidence pack must export one immutable receipt joining:

  • Activity trace — the full step-by-step actions the agent took (prompt + tool-call + retrieval + edit + commit), with timestamps + per-step latency + failure mode if any.
  • Source citations — every Knowledge Graph source object the agent cited, with object ID + checksum + retrieval-window + access-policy at retrieval-time.
  • Decision log — the branching decisions the agent made (which path it picked + why + what alternatives it rejected), with the policy rule version that governed each branch.
  • Connected-app permissions — which SaaS connectors were active during the run + what permission scope each was authorized under + when the authorization was last confirmed.
  • Human approvals — the human-in-the-loop approval events captured before any side-effect-producing action (publish + send + commit + purchase), with approver identity + timestamp + approval rationale.
  • Tenant identity — the org + workspace + user that owned the run, with the RBAC role + SSO identity-provider + MFA state at run-time.
  • Control versions — the model version (Cascade 1) + guardrail version + connector version + policy version + retention-rule version that were active for the run, all pinned.

This is the join EU AI Act Article 14(4) requires for high-risk AI systems with autonomous action — provable human-oversight operational record per-tenant per-action.

Cascade 3 — Knowledge Graph retrieval → source object + index version + checksum + access state

For every material answer an agent produced from Knowledge Graph (Graph RAG) retrieval, the evidence pack must export one immutable receipt joining:

  • Source object — the exact document + object + row + paragraph the agent cited, with the source object's identity in the WRITER Knowledge Graph namespace.
  • Index version — the Graph RAG index version active at retrieval-time, with index-build-date + embedding-model-version + chunking-config-version.
  • Checksum — SHA-256 of the source object content at retrieval-time, so the auditor can verify the cited text has not been altered since retrieval.
  • Access state — the source object's permission + retention + deletion state at retrieval-time — was the actor authorized to read it? Was it within the retention window? Had any deletion-trigger fired?
  • Permissions — the user/role/group authorization at retrieval-time, with the policy version that governed the authorization decision.

This is the join that proves "the agent retrieved source X with index Y and produced answer Z" without forcing the auditor to chase the WRITER Knowledge Graph admin console.

Cascade 4 — policy / connector / retention change → affected workflows + preserved prior state

When a model, guardrail, policy, connector, role, or retention setting changes, the evidence pack must preserve the prior state and identify affected agent workflows instead of replacing the old evidence. Per-change immutable receipt joining:

  • Change subject — what changed (model version + guardrail version + policy version + connector version + RBAC role + retention rule + tenant setting).
  • Prior state — the value before the change, with the timestamp the prior value became active.
  • New state — the value after the change, with the timestamp the new value became active.
  • Affected workflows — the WRITER Agent workflows + Knowledge Graph queries + customer-facing tool-calls that were using the prior state at the time of change.
  • Change rationale — the owner + ticket + change-management-record identifier that approved the change.
  • Re-evaluation evidence — the re-evaluation + re-test + re-customer-notification records that were produced in response to the change (EU AI Act Article 14(4) post-market-monitoring).

This is the join ISO/IEC 42001 AI management system + SOC 2 Type II change-management controls + GDPR Art. 28(2) sub-processor-change-notification SLA all require.

Cascade 5 — machine-readable bundle → assertion → first-party evidence + owner + collection time + exception + remediation status

Procurement must be able to retrieve the result as one machine-readable bundle (JSON-LD + signed PDF + audit-trail CSV) that maps every assertion in the pack to:

  • First-party evidence — the source URL + signed receipt + immutable hash + retrieval timestamp.
  • Owner — the named individual or team accountable for the assertion.
  • Collection time — when the evidence was collected + when the receipt was sealed.
  • Exception — any gap, deviation, control-failure, or out-of-period condition that affected the assertion.
  • Remediation status — ticket + owner + ETA + state (open / accepted / closed) for any exception.
  • Control mapping — the control IDs from EU AI Act + ISO/IEC 42001 + GDPR + SOC 2 Type II + sector review the assertion supports.

This is the join that turns a WRITER trust-center page (which is a marketing surface) into a procurement-grade evidence artifact (which is an audit surface). Same ingredients, different shape.
