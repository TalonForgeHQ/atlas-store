Gap 1 — Questionnaire intake and schema provenance

For each Whistic Assess questionnaire intake, preserve the request source (Whistic Assess direct, Trust Catalog, Trust Center Exchange, inbound customer), questionnaire schema version (SIG Lite / CAIQ / VSAQ / NIST 800-161 / ISO 27001 / SOC 2 / custom), question-mapping version, vendor profile, reviewer assignment, and intake timestamp.

  • Intake source + questionnaire schema version
  • Question-mapping version + framework mapping (SIG / CAIQ / ISO / SOC 2 / NIST AI RMF / EU AI Act)
  • Vendor profile + reviewer assignment
  • Intake timestamp + customer-visible status

Gap 2 — Whistic AI questionnaire response generation chain

For each AI-generated questionnaire answer, preserve the prompt version, LLM sub-processor version + region + retention window, generated answer draft, reviewer-in-the-loop accept/edit/reject event, finalized answer, and versioned response object. This is the EU AI Act Article 14(4) human-oversight + Article 50 transparency + Article 53(1)(b) GPAI cascade that procurement teams will request verbatim.

  • Whistic AI prompt version + LLM sub-processor version
  • Region + retention window + training-data opt-out flag
  • Reviewer-in-the-loop accept/edit/reject event + timestamp
  • Finalized answer + response-object hash + downstream publication

Gap 3 — Trust Catalog + Trust Center Exchange publication chain

For each Trust Center artifact published through Whistic Profile, Whistic Trust Catalog, or Whistic Trust Center Exchange, preserve the vendor feed source, published-artifact schema version, exposure-tier setting (public / NDA-gated / private), customer-visible delta notification, audit-log hash, and Exchange-side viewer access log.

  • Vendor feed source + published-artifact schema version
  • Exposure-tier setting + customer-visible delta notification
  • Audit-log hash + Exchange-side viewer access log
  • Publication timestamp + revision lineage

Gap 4 — Vendor monitoring + reassessment cadence

For each vendor on the monitoring surface, preserve the vendor-criticality tier, monitoring source (Whistic Profile feed, external scan, SIG update, CAIQ update, customer-supplied event), reassessment trigger, reviewer assignment, evidence pack refresh, and customer-visible status change. Without these, a buyer cannot answer "why was this vendor's tier raised?" or "what changed since the last review?"

  • Vendor-criticality tier + monitoring source
  • Reassessment trigger + reviewer assignment
  • Evidence pack refresh + delta summary
  • Customer-visible status change + timestamp

Gap 5 — Access, retention, deletion, and policy/model change evidence

For each access, correction, retention, deletion, or policy/model change request, identify every affected tenant, region, vendor assessment, Exchange entry, Trust Center artifact, Whistic AI conversation, provider-held data, cache, backup, and the reproducible export hash a customer can use as audit evidence.

  • Request type + tenant/region + timestamp
  • Affected vendor assessments + Exchange entries + Trust Center artifacts
  • Reprocessed model/policy version + delta
  • Export hash + retention/deletion receipt
