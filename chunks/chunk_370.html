1. Per-AI-action provenance across Figma AI + Make + Dev Mode + plugin ecosystem

Figma AI now produces frames, components, content rewrites, asset generations, auto-layout suggestions, smart components, image generations, and Make-output designs via the prompt-to-prototype pipeline. Each mutation needs immutable evidence of which prompt + model + version produced it, which tool calls were invoked, which files/frames/components/styles were mutated, and which downstream integrations received the change. The current audit trail only records the file edit, not the AI decision chain. Specifically:

  • Figma AI frame generations — prompt/model/version + tool call sequence
  • Make-output generations — design-to-code prompt + response + file mutation
  • AI-replaced content blocks — model + context-window + output
  • Dev Mode code handoffs — code generation + VS Code sync + read receipt
  • FigJam AI summaries + clusters + brainstorms — prompt + frame-level output
  • Plugin writes — plug-in identity + sandbox-state + tool call + write target
  • External integration writes — Jira/Linear/Notion/Asana/GitHub payload + propagated record-id + propagation timestamp

2. Workspace/team/file/component/library/data-residency isolation

Figma's collaborative canvas + Dev Mode + the Figma plug-in ecosystem operate across enterprise and team plans, US and EU data residency, SAML/SCIM-authenticated members and guest commenters, personal draft workspaces and shared team files, plug-in sandboxes and the underlying file API. The isolation matrix must be airtight on every transition:

  • Enterprise vs team plan boundary — billing-tier + capability-tier isolation
  • US vs EU data residency — storage + processing + transit + encryption boundary
  • SAML/SCIM member vs guest commenter — privilege + read/write boundary
  • Personal draft workspace vs shared team file — membership + lock state
  • Plug-in sandbox vs underlying file API — capability + network + persistence boundary
  • Branch vs main file — merge provenance + branch state + read access

3. Prompt-injection defense for untrusted content

Figma files regularly import text from Slack, Teams, Google Docs, Jira tickets, Linear issues, Notion pages, Asana tasks, GitHub PR bodies, and customer-submitted component descriptions. That untrusted text becomes part of the file body that Figma AI reads. The defense playbook needs to cover prompt injection through all of these channels:

  • Slack/Teams messages → Figma file body → Figma AI context window
  • Jira ticket body → synced to Figma comment → Figma AI read
  • Linear issue body → synced to Figma → Figma AI read
  • Notion page → embed into Figma frame → Figma AI read
  • GitHub PR body → Figma issue-tracker sync → Figma AI read
  • Customer-submitted component descriptions → Figma file body → Figma AI read
  • Plugin-rendered content → Figma file body → Figma AI read

4. Deletion/retention/rollback/branch-merge/integration-sync propagation

When a user deletes a frame, branch-merges a prototype, or integrates with Jira/Linear/Notion/Asana/GitHub, the propagation to the connected surfaces needs to be auditable with retention windows and rollback capability:

  • Frame deletion → version-history retention + branch visibility + audit log

  • GitHub PR merge → Figma branch state + read receipt
