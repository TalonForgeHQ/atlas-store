The five evidence questions

1. Can every completion be reconstructed?

A completion_receipt should join completion ID, tenant, developer identity, IDE/plugin version, proprietary model version, prompt-policy version, source-context hash, deployment mode, timestamp, output hash, and human acceptance or rejection. Without that join, “private model” remains a platform-level claim rather than event-level evidence.

2. Can every deployment mode prove its boundary?

The evidence pack should pin secure SaaS, VPC, on-premises, or air-gapped mode to region, network boundary, customer-controlled key or approved key reference, model artifact checksum, configuration checksum, data-egress policy, administrator approval, and deployment timestamp.

3. Can zero code retention be verified per request?

For cloud processing, a buyer needs a retention receipt that joins request ID, ephemeral-processing state, policy version, deletion or expiry event, storage classes touched, exception state, and verifier. “Zero retention” becomes auditable only when the system can prove that a request did not cross into logs, training datasets, backups, or support exports.

4. Can no-training policy be traced into model releases?

A training_exclusion_attestation should link customer tenancy and policy state to dataset admission rules, data-source inventory, training-run ID, exclusion check, reviewer, model release, and exception status. This is the control that turns “we never train on your code” into evidence a buyer can test.

5. Can an air-gapped customer update and roll back safely?

Offline customers need a signed update bundle containing model and application artifacts, SBOM, vulnerability scan, signing certificate, compatibility matrix, administrator approval, installation receipt, health check, and rollback receipt. Air-gapping reduces network exposure but raises the importance of release provenance.

Procurement map

  • Software supply chain: signed model/application artifacts, SBOM, vulnerability scan, installation receipt, rollback proof, and administrator approval.
  • Privacy: data-flow inventory, storage-class inventory, zero-retention receipt, no-training attestation, tenant isolation, and deletion exception handling.
  • AI governance: model version, prompt-policy version, output provenance, human disposition, evaluation result, change approval, and release owner.
  • Access control: developer identity, tenant, role, IDE session, administrative change, deployment approval, and emergency-access event.
  • Operational resilience: offline compatibility, update cadence, rollback path, backup posture, incident receipt, and evidence-export availability.
