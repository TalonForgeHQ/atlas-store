Five buyer-facing joins

  1. Data asset → governed context → output: source asset ID, catalog or policy version, retrieval or feature context, model/agent version, output hash, citation/source reference, tenant, and region.
  2. Build → evaluation → release: source/code revision, Agent Bricks or Mosaic AI configuration, model/version, evaluation dataset, metric, reviewer, decision, deployment revision, rollback target, and release timestamp.
  3. Agent → tool/data action → authorization: tenant, user or service identity, requested tool or data scope, catalog permission, policy version, approval, downstream state change, denial, retry, and credential boundary.
  4. Human oversight → remediation: reviewer identity, prompt/model/policy change, reason, expected behavior, observed behavior, severity, corrective action, approval decision, and production release event.
  5. Export → deletion: source tables, catalog metadata, embeddings or feature artifacts, prompts, traces, evaluation artifacts, caches, logs, replicas, backups, completion time, exception, and sub-processor dependency.

Common evidence failures

  • Treating “we use Databricks” as proof of the exact workspace, catalog, policy, model, and agent boundary.
  • Showing an evaluation score without the dataset version, reviewer decision, promoted revision, or rollback target.
  • Logging a tool call without the permission decision, credential boundary, downstream state change, and human escalation path.
  • Proving deletion of a table while omitting prompts, traces, caches, evaluation artifacts, replicas, or backups.
