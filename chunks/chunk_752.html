Long-tail target: Daytona stateful AI sandbox audit checklist, with supporting searches for Daytona snapshot provenance, Daytona isolated AI code execution evidence, Daytona persistent volume audit, Daytona computer-use sandbox receipts, and Daytona customer-managed compute compliance.

Daytona's current first-party product material positions the platform as secure infrastructure for running AI-generated code and as fast, scalable, stateful infrastructure for AI agents. The public surface includes sub-90ms sandbox creation, isolated execution, snapshots, persistent sandboxes, volumes, process/file/Git/LSP APIs, multi-region operation, computer-use environments, human-in-the-loop access, and customer-managed compute. The buyer question is whether a material agent run can be reconstructed after the snapshot, runtime, region, files, credentials, tools, people, and external systems have changed.

Gap 1 — Environment, image, and snapshot lineage

Every run should bind the customer tenant, sandbox ID, image or Docker input, snapshot version, runtime and SDK versions, operating system, deployment region, and customer-managed-compute identity into one immutable receipt.

  • Sandbox, image, snapshot, runtime, SDK, and operating-system identifiers
  • Region, tenant, compute-plane, and customer-key-scope identities
  • Creation, restore, resume, pause, expiration, and deletion timestamps
  • Snapshot checksum plus the policy and actor that promoted it

Gap 2 — Process, code, file, Git, and LSP provenance

Programmatic control becomes auditable when every process execution, code run, file mutation, Git operation, and language-server event carries a causal parent and verifiable before-and-after state.

  • Agent, model/provider, prompt, tool, and policy versions
  • Process, command, code, file, Git, and LSP event IDs
  • Input/output hashes, working directory, permissions, exit code, and duration
  • Resulting filesystem, repository, dependency, and external-state hashes

Gap 3 — Persistent volume and shared-data boundary

A persistent agent may outlive one process or sandbox. Each volume attach, read, write, snapshot, restore, and detach event should preserve tenant scope, authorization, data classification, and a before-and-after hash.

  • Volume, tenant, sandbox, mount, and data-classification identities
  • Attach/detach authorization, access mode, and policy version
  • File delta, snapshot, restore, retention, and cross-sandbox reuse events
  • Deletion cascade across active mounts, snapshots, replicas, and backups

Gap 4 — Computer use and human-intervention receipts

Daytona publishes Linux, Windows, and macOS computer-use environments plus SSH, VS Code Browser, and Web Terminal access. A reviewer should be able to separate autonomous actions from operator intervention and tie either path to the external state it changed.

  • Desktop OS/image, session, agent, model, tool, and target-state versions
  • Mouse, keyboard, shell, browser, application, and network action IDs
  • SSH, VS Code, or Web Terminal actor, authentication, and access scope
  • Human approval, override, correction, handback, and downstream receipt

Gap 5 — Reproducibility, rollback, retention, and deletion

A snapshot is useful to assurance teams only if it can reproduce a run and prove what did not survive teardown. Retain the replay result, divergence report, rollback state, process termination, volume disposition, log expiry, and deletion completion.

  • Snapshot replay artifact, expected-state hash, and observed-state hash
  • Dependency, region, runtime, network, clock, and secret-scope divergence
  • Rollback, retry, failure-recovery, and human-remediation events
  • Sandbox, process, volume, log, artifact, replica, and backup deletion receipt

Five-question procurement test

  1. Can one Daytona run be reconstructed from image, snapshot, runtime, OS, region, tenant, and compute-plane identity?
  2. Can a reviewer trace one agent instruction through process, code, file, Git, LSP, output, and resulting-state events?
  3. Can every persistent-volume attach, mutation, restore, reuse, and deletion event be tied to tenant scope and policy?
  4. Can autonomous computer-use actions be separated from SSH, VS Code, or Web Terminal intervention and tied to downstream side effects?
  5. Can a snapshot replay, rollback, teardown, retention expiry, and deletion cascade be exported as one dated receipt?
