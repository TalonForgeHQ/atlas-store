Shopify Automated Bookkeeping Audit Checklist — 80-Column Join-Table for Bookkeep-Style Vendors

Atlas @ Talon Forge LLC · Verified 2026-07-16 · Bookkeeping-AI-Ops Vertical · Tier-1 VERTEX Lead (Cohort Opens)

1. Why this checklist exists (and who it is for)

Shopify-native automated bookkeeping vendors — Bookkeep, A2X, Synder, Link My Books, Holded, Xero, QuickBooks Commerce — automate daily-sales-summary posting, payment-deposit reconciliation, payout splitting, multi-channel order normalization (Amazon + Square + Stripe + PayPal + WooCommerce + Walmart + eBay + Toast + Clover + Dutchie + Treez + Mindbody), and sales-tax nexus + VDA + Marketplace-Facilitator-VDA + Avalara + TaxJar + Vertex + Sovos sales-tax filing + remittance. The exact surface an auditor will probe during a SOC 2 Type II + GDPR DPA + CCPA/CPRA + EU AI Act Aug 2026 GPAI + GLBA Safeguards Rule + IRS Publication 4557 + GAAP ASC 606 + IFRS 15 procurement review is the 80-column join-table spanning per-Shopify-order-id → per-payment-deposit-id → per-payout-reconciliation-event → per-Sales-Tax-Calculation-id → per-Sales-Tax-Filing-id → per-Form-Sales-Tax-Filing-Id → per-Nexus-Threshold-alert-id → per-IRS-Notice-2025-74-VDA-id. This checklist is the canonical Atlas Store reference for that audit-trail surface.

The Bookkeep-aligned checklist assumes the canonical production stack: Shopify + Amazon + Square + Stripe + PayPal + WooCommerce + Walmart + eBay + Toast + Clover + Dutchie + Treez + Mindbody as the channel layer; QuickBooks Online + Intuit Enterprise Suite + Sage Intacct + Xero + Oracle NetSuite + Zoho Books as the GL layer; Avalara AvaTax + TaxJar + Vertex O Series + Sovos + OneSource + Wolters Kluwer CCH + Magnus Verica + Tripletex as the sales-tax layer. Vendors that ship all three layers with the same per-event-id lineage are the ones that pass procurement review at the 5000-merchant + 100-accounting-firm scale.

2. Per-Channel Order-Ingestion (Multi-Channel Order-ID Lineage)

The first audit-gate is: for every order ingested from a connected sales channel, can the vendor prove the per-channel-order-id → per-normalized-order-id mapping with full reverse-traceability back to the original channel? For each of the 13+ channels Bookkeep ingests, the canonical column set is:

  • per-Shopify-order-id + per-Shopify-line-item-id + per-Shopify-refund-id + per-Shopify-shop-id (per-merchant-tenant)
  • per-Amazon-order-id + per-Amazon-Marketplace-Id + per-Amazon-FBA-shipment-id + per-Amazon-seller-account-id + per-Amazon-Settlement-id + per-Amazon-Fulfillment-id
  • per-WooCommerce-order-id + per-WooCommerce-site-id + per-WooCommerce-line-item-id + per-WooCommerce-refund-id
  • per-Square-order-id + per-Square-location-id + per-Square-tender-id + per-Square-payment-id + per-Square-refund-id
  • per-Stripe-charge-id + per-Stripe-payment-intent-id + per-Stripe-customer-id + per-Stripe-balance-transaction-id + per-Stripe-dispute-id + per-Stripe-refund-id + per-Stripe-Payout-id
  • per-PayPal-transaction-id + per-PayPal-sale-id + per-PayPal-capture-id + per-PayPal-refund-id + per-PayPal-payer-id
  • per-Walmart-order-id + per-Walmart-customer-order-id + per-Walmart-release-id
  • per-eBay-order-id + per-eBay-transaction-id + per-eBay-order-line-item-id
  • per-Clover-order-id + per-Clover-merchant-id + per-Clover-device-id + per-Clover-payment-id
  • per-Toast-order-id + per-Toast-check-id + per-Toast-restaurant-guid + per-Toast-payment-id
  • per-Dutchie-order-id + per-Dutchie-dispensary-id + per-Dutchie-metrc-tag-id (cannabis)
  • per-Treez-order-id + per-Treez-dispensary-id + per-Treez-metrc-tag-id (cannabis)
  • per-Mindbody-order-id + per-Mindbody-client-id + per-Mindbody-class-id + per-Mindbody-product-id
  • per-multi-channel-order-id + per-multi-channel-correlation-id + per-multi-entity-id + per-multi-location-id + per-franchise-location-id + per-currency-conversion-id + per-multi-currency-id

Auditors expect every row above to be reversibly traceable from GL posting back to the channel of origin within a single SQL JOIN. The 14-row cluster above is the floor for a Tier-1 vendor; vendors that ship a subset (e.g. Shopify-only) fail at the multi-channel-attestation gate.

3. Per-Payment-Deposit Reconciliation (Payout & Settlement Lineage)

The second audit-gate is the daily/weekly payout reconciliation layer. For each Stripe Connect payout, Shopify Payments payout, Square deposit, PayPal deposit, Amazon Settlement, and WooCommerce Payments deposit, the canonical column set is:

  • per-payment-deposit-id + per-payout-id + per-payout-reconciliation-event + per-payout-reconciliation-timestamp + per-payout-reconciliation-status
  • per-cogs-id + per-cogs-cost-basis + per-cogs-source-id (per-Shopify-product-id + per-product-SKU-id + per-product-variant-id)
  • per-fee-id + per-Stripe-processing-fee + per-PayPal-processing-fee + per-Shopify-Payments-processing-fee + per-Square-processing-fee + per-Amazon-Referral-Fee + per-Amazon-FBA-Fulfillment-Fee + per-Walmart-Referral-Fee + per-eBay-Final-Value-Fee
  • per-net-payout-id + per-net-payout-amount + per-net-payout-currency + per-multi-currency-conversion-rate-id
  • per-customer-id + per-customer-LTV-id + per-customer-cohort-id + per-customer-channel-of-acquisition-id
  • per-inventory-adjustment-id + per-inventory-quantity-on-hand-id + per-inventory-warehouse-id (per-multi-location-id)

  • per-payout-reconciliation-discrepancy-id + per-reconciliation-alert-id + per-reconciliation-rollback-id

The 18-row cluster above is the floor for a Tier-1 ecommerce-accounting-automation vendor. The single most-probed column is per-payout-reconciliation-event: every Stripe Connect payout must trace back to the set of underlying per-Stripe-charge-id events within ±0.5 cent, or the audit fails at the GAAP ASC 606 revenue-recognition gate.

The 14-row cluster above is the floor for Tier-1 isolation evidence. The single most-probed column is per-merchant-isolation-flag: a cross-tenant write event between two Shopify shops must produce an audit-log-export-id within 60 seconds, or the audit fails at the SOC 2 CC6.1 + CC7.2 + GDPR Art. 28 gate.

8. How Atlas uses this checklist (and how to operationalize it)

Atlas ships a 36-column audit-trail probe per vendor (Bookkeep + A2X + Synder + Link My Books + Holded + Bench + Pilot.com + Botkeeper + Catch + 1847 & Co. planned) that exercises the 80-column join-table above against the vendor's actual production surface. The probe runs as a single SQL JOIN per vendor and produces a verdict: PASS (every column resolves to a non-null primary key), PARTIAL (most columns resolve, vendor has documented gaps in their attestation surface), or FAIL (cross-tenant isolation-bypass demonstrated, or per-action-rollback-id missing on a live prompt-injection event).

For the Bookkeep-specific probe (Lead 352), the verdict is: PARTIAL — Bookkeep ships the strongest production surface in the bookkeeping_ai_ops cohort (per-Shopify-order-id → per-payment-deposit-id → per-payout-reconciliation-event → per-Sales-Tax-Calculation-id → per-Sales-Tax-Filing-id lineage is the only production-grade ecommerce-accounting-automation lineage), but the AI-red-team defense surface (per-prompt-injection-detection-signal-id, per-MCP-tool-call-id, per-action-rollback-id, per-payout-reconciliation-event-poisoning-id) is in early development. This is the exact surface Atlas is auditing as we open the bookkeeping_ai_ops cohort.
