Modal's current first-party product material positions the platform as high-performance AI infrastructure: bring-your-own code, run CPU, GPU, and data-intensive compute at scale, scale from zero to thousands of CPUs or GPUs in a few lines of code, $30/month free compute, the modal Python SDK, the modal run CLI, Modal Sandboxes with automated health checks and readiness probes, Inference endpoints, Training jobs, Notebooks, Batch, and the Core Platform. The buyer question is whether a material agent run can be reconstructed after the sandbox ID, image, snapshot, runtime, region, files, secrets, tools, people, and downstream systems have changed.

Gap 1 — Serverless sandbox, image, and snapshot lineage

Modal exposes Function and Sandbox identities that should bind the customer tenant, sandbox or function ID, image or Docker input, Memory Snapshot version, Filesystem and Directory Snapshot version, runtime and SDK versions, region, and customer-key-scope into one immutable receipt.

  • Sandbox / function / image / snapshot / runtime / SDK / operating-system identifiers
  • Region, tenant, compute-plane, and customer-key-scope identities
  • Memory Snapshot + Filesystem and Directory Snapshot creation, restore, resume, expiration, and deletion timestamps
  • Image rebuild checksum plus the policy and actor that promoted it

Gap 2 — Execution, function call, and filesystem provenance

Modal's serverless surface routes through Functions, Inference endpoints, and Modal Sandboxes. Programmatic control becomes auditable when every process execution, code run, file mutation, function call, image rebuild, and volume event carries a causal parent and a verifiable before-and-after state.

  • Agent, model or provider, prompt, tool, and policy versions
  • Function input, output, latency, exit code, cold-start, and concurrency identifiers
  • Inference endpoint call, request/response hash, and resulting filesystem or downstream state
  • File, image, function, and Volume event IDs with input/output hashes and resulting state

Gap 3 — Volume boundary and HIPAA compliance

Modal publishes Volumes v1 (out-of-HIPAA scope) and HIPAA-Compliant Volumes v2. Each Volume mount, read, write, snapshot, restore, and detach event should preserve tenant scope, HIPAA-or-non-HIPAA classification, and a before-and-after hash.

  • Volume v1 vs Volume v2 identity, tenant, sandbox, mount, and data-classification
  • Attach / detach authorization, access mode, and policy version
  • File delta, snapshot, restore, retention, and cross-sandbox reuse events
  • Deletion cascade across active mounts, snapshots, replicas, backups, and BAA-relevant PHI

Gap 4 — Readiness probes, health checks, and human intervention

Modal publishes automated health checks, readiness probes (TCP port / file / any shell condition), detailed logging and metrics, observability dashboards, and the Modal Sandbox dashboard control plane. A reviewer should be able to separate autonomous actions from operator intervention and tie either path to the resulting state.

  • Sandbox / function / agent / model / tool / target-state versions
  • Readiness probe definition, health-check signal, and observed-state diff
  • Operator, viewer, debugger, and CLI-actor authentication and access scope
  • Human approval, override, correction, handback, and downstream receipt

Gap 5 — Reproducibility, retention, deletion, and shared responsibility

Modal's shared-responsibility model assigns customers backups, recovery, availability, and security measures while Modal owns platform integrity, security, and availability. A snapshot is useful to assurance teams only if it can reproduce a run and prove what did not survive teardown. Retain the replay result, divergence report, rollback state, process termination, volume disposition, log expiry, and deletion completion.

  • Snapshot replay artifact, expected-state hash, and observed-state hash
  • Dependency, region, runtime, network, clock, and secret-scope divergence
  • Rollback, retry, failure-recovery, and human-remediation events
  • Sandbox, function, volume, log, artifact, replica, backup, and BAA deletion receipt

Five-question procurement test

  1. Can one Modal run be reconstructed from sandbox / function / image / Memory Snapshot / Filesystem Snapshot / runtime / region / tenant / customer-key identity?
  2. Can a reviewer trace one agent instruction through function call, Inference endpoint, file mutation, image rebuild, Volume event, and resulting-state events?
  3. Can every Volume v1 / Volumes v2 attach, mutation, restore, reuse, and deletion event be tied to tenant scope, HIPAA-or-non-HIPAA classification, and policy?
  4. Can autonomous actions be separated from operator intervention and tied to the downstream side effects they changed?
  5. Can a snapshot replay, rollback, teardown, retention expiry, and deletion cascade be exported as one dated receipt, with the shared-responsibility boundary explicitly tagged?
