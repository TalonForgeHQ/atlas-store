1. The 24 Rossum-specific audit-gaps (grouped by evidence surface)

1.1 Per-document-page-lineage (5 audit-gaps)

  1. Per-document-id persistence — must be unique + immutable + cross-system-joined (document → page → line-item → OCR-token → field → LLM → reviewer → correction → ERP-write).
  2. Per-page-id + per-page-hash — every page of every document gets a stable page-id + cryptographic hash. Auditor wants per-page-original-storage-URL + per-page-OCR-source-version + per-page-OCR-cache-TTL.
  3. Per-line-item-id + per-line-item-bounding-box — Rossum's Aurora AI engine extracts each line item (invoice-line, PO-line, BOL-line, customs-line) with a JSON bounding-box coordinate. Auditor wants per-bounding-box-x + per-bounding-box-y + per-bounding-box-width + per-bounding-box-height + per-bounding-box-confidence.
  4. Per-OCR-token-id + per-OCR-confidence-score — every OCR token (character, word, sub-word) gets a token-id + a confidence-score between 0 and 1. Auditor wants per-token-source-OCR-engine + per-token-OCR-engine-version + per-token-confidence-decision-threshold.
  5. Per-OCR-confidence-score aggregation — Rossum aggregates per-token confidence into per-field, per-line-item, per-page, per-document confidence. Auditor wants per-aggregation-formula + per-aggregation-threshold-policy.

1.2 Per-LLM-and-prompt-lineage (5 audit-gaps)

  1. Per-LLM-model-id + per-LLM-prompt-hash — which model extracted which field (Claude Sonnet, GPT-4o, Rossum Aurora fine-tuned, customer fine-tuned) + which prompt-template (Rossum ships prompt-template-versioning as an Aurora primitive).
  2. Per-prompt-template-version — Rossum's prompt-template library is a first-class export surface. Must include template-hash + template-owner + template-review-status + template-effective-date + template-deprecation-policy.
  3. Per-field-extraction-id + per-field-validation-id — Rossum pairs every extracted field with a stable extraction-id (LLM + OCR + rules-engine consensus) AND a validation-id (schema + business-rules + ERP-validation). Auditor wants per-extraction-engine-composition + per-validation-rule-version.
  4. Per-prompt-injection-detection-signal-id — Rossum ships prompt-injection-detection as a runtime guard on Aurora. Auditor wants per-injection-attempt-attribution + per-injection-mitigation-action (block, sanitize, escalate-to-human, log-and-continue) + per-injection-false-positive-rate-tracking.
  5. Per-data-residency-region — Rossum operates EU + UK + US regions. Auditor wants per-region data-residency-policy + per-region subprocessor-list + per-region cross-border-transfer-mechanism (SCCs / DPF / EU-US Data Privacy Framework).

1.3 Per-human-in-loop-and-correction-lineage (5 audit-gaps)

  1. Per-human-reviewer-id + per-review-decision — every line item a human touches (approve, reject, correct, escalate) is attributed. Auditor wants per-reviewer-authentication-method + per-reviewer-authorization-role + per-reviewer-SSO-identity-provider.
  2. Per-correction-id + per-correction-version — every human correction is recorded with original-value + corrected-value + correction-reason + correction-timestamp + correction-reviewer-id. Auditor wants per-correction-rollback-capability + per-correction-audit-log-immutability.
  3. Per-rollback-id — Rossum supports per-document and per-line-item rollback when a downstream ERP / SAP / Coupa write fails. Auditor wants per-rollback-trigger + per-rollback-approver + per-rollback-state-recovery.
  4. Per-resolution-quality-score — Rossum ships a resolution-quality-score that combines OCR-confidence + LLM-confidence + reviewer-disagreement-rate + downstream-ERP-validation-success. Auditor wants per-resolution-score-formula + per-resolution-score-threshold-policy.
  5. Per-AI-action-id + per-AI-action-rollback-id — every write the Aurora engine initiates (ERP-write, SAP-post, Coupa-push, NetSuite-commit) is attributed + reversible. Auditor wants per-write-target-system + per-write-payload-version + per-write-rollback-SLA.

1.4 Per-action-and-security-attestation (4 audit-gaps)

  1. Per-AI-action-type + per-AI-action-target — Rossum Aurora classifies each AI action (approve-invoice, post-gl-entry, push-to-coupa, match-PO-line, etc.). Auditor wants per-action-type-policy + per-action-type-approval-workflow.
  2. Per-AI-action-rollback-reason — when a write is rolled back, Rossum requires a reason-code (validation-fail, fraud-signal, reviewer-override, ERP-reject, duplicate-detected). Auditor wants per-reason-code-mapping + per-reason-code-incident-response-SLA.
  3. Per-prompt-injection-detection-signal-coverage — auditor wants per-prompt-coverage (what % of Rossum's Aurora prompt surface is covered by injection-detection) + per-injection-block-rate + per-injection-bypass-rate.
  4. Per-cross-border-transfer-mechanism — when data crosses region (EU → US, UK → US), auditor wants per-transfer-mechanism (SCCs-2021/914, DPF, UK-IDTA, EU-US Data Privacy Framework) + per-transfer-impact-assessment.

1.5 Per-compliance-and-subprocessor-attestation (5 audit-gaps)

  1. Per-subprocessor-list-version — Rossum publishes a subprocessor list at rossum.ai/subprocessors/. Auditor wants per-subprocessor-name + per-subprocessor-role + per-subprocessor-region + per-subprocessor-data-category.
  2. Per-DPA-version + per-DPA-signature-date — Rossum signs per-customer DPAs. Auditor wants per-DPA-template-version + per-DPA-signed-date + per-DPA-effective-date + per-DPA-termination-clause.
  3. Per-SOC-2-Type-II-report-period — Rossum publishes SOC 2 Type II annually. Auditor wants per-report-period-start + per-report-period-end + per-auditor-name + per-report-Coverage-CC + per-report-exceptions-list.
  4. Per-ISO-27001-certificate-id — Rossum is ISO 27001 certified. Auditor wants per-certificate-id + per-cert-issuing-body + per-cert-issue-date + per-cert-expiry-date + per-cert-scope-statement.
  5. Per-EU-AI-Act-Article-14-readiness-evidence — Rossum is on the EU AI Act readiness path. Auditor wants per-Article-14-human-oversight-evidence + per-Article-15-accuracy-evidence + per-Article-53(d)-log-evidence + per-Article-9-risk-management-evidence.

2. The 14-question buyer checklist Rossum's procurement team gets sent

  1. Provide the schema (DDL or JSON-schema) for the 24 audit-gaps above with sample values from a redacted production document.
  2. Confirm per-document-id + per-page-id + per-line-item-id immutability and cross-system join semantics across OCR + Aurora + reviewer + ERP-write.
  3. Confirm per-OCR-token-id retention window + per-OCR-confidence-score aggregation formula + per-OCR-engine-version-control.
  4. Confirm per-LLM-model-id + per-LLM-prompt-hash coverage — what % of Aurora extraction calls have a model-id + prompt-hash, and what's the fallback behavior when the model is unidentifiable?
  5. Confirm per-prompt-template-version governance — who owns, who reviews, who approves, who deprecates, and what's the customer-side override mechanism?
  6. Confirm per-field-extraction-id + per-field-validation-id persistence and per-customer export mechanism (S3, GCS, webhook, Snowflake, BigQuery).
  7. Confirm per-human-reviewer-id + per-review-decision SSO integration (Okta, Azure AD, Google Workspace, Ping, Auth0) and per-reviewer-action audit-log immutability.
  8. Confirm per-correction-id + per-rollback-id reversibility — what is the per-rollback-SLA when an ERP-write fails?
  9. Confirm per-resolution-quality-score formula transparency — can the customer recalculate the score from per-extraction-id + per-reviewer-id + per-ERP-write-success-rate?
  10. Confirm per-AI-action-id + per-AI-action-rollback-id + per-AI-action-rollback-reason coverage — what % of Aurora writes have a rollback-reason code attached?
  11. Confirm per-prompt-injection-detection-signal-id detection-coverage, per-detection-block-rate, and per-detection-bypass-rate across Aurora's full prompt surface.
  12. Confirm per-data-residency-region pinning — can an EU customer guarantee no US-region subprocessor is invoked for any Aurora call? Provide the per-subprocessor-region matrix.
  13. Confirm per-cross-border-transfer-mechanism for EU → US flows post-Coupa-acquisition (SCCs-2021/914 vs DPF vs UK-IDTA) and provide the transfer-impact-assessment.
  14. Confirm per-SOC-2-Type-II-report-period coverage of the 24 audit-gaps above and provide per-CC7.2 + per-CC8.1 evidence trail for the last 12 months.

3. 6-layer reference architecture (where the 24 audit-gaps are produced)

  1. Ingest layer — document receipt (email, SFTP, S3, Coupa inbox, SAP inbox, NetSuite inbox, Workday inbox, custom-API). Produces per-document-id + per-page-id + per-page-hash.
  2. OCR layer — Rossum's proprietary OCR + Aurora LLM consensus. Produces per-line-item-id + per-line-item-bounding-box + per-OCR-token-id + per-OCR-confidence-score.
  3. Extraction layer — Aurora LLM + prompt-template-versioning. Produces per-LLM-model-id + per-LLM-prompt-hash + per-prompt-template-version + per-field-extraction-id.
  4. Validation layer — Rossum's schema-validation + business-rules-engine + ERP-validation. Produces per-field-validation-id + per-validation-rule-version + per-resolution-quality-score.
  5. Reviewer layer — human-in-loop UI with SSO + per-human-reviewer-id + per-review-decision + per-correction-id. Produces the per-correction-version + per-rollback-id trail.
  6. Write layer — ERP-write + SAP-post + Coupa-push + NetSuite-commit + Workday-push. Produces per-AI-action-id + per-AI-action-type + per-AI-action-rollback-id + per-AI-action-rollback-reason.

Per-human-reviewer-id SSO + per-correction-id persistence is first-class — Rossum's reviewer UI pairs SSO identity (Okta/Azure AD/Google Workspace/Ping) with per-review-decision attribution + per-correction-version rollback.
