Why a serverless AI runtime needs its own evidence chain

A production AI application on Modal does not move directly from a developer call to a result. A function call enters the Modal runtime, schedules onto a container pool, optionally mounts a GPU, may launch a Modal Sandbox, executes the user's code, captures memory snapshots on a schedule or on OOM, and writes results back through a Webhook, Schedule, or Queue. If a security reviewer later asks why the system produced a particular output, executed untrusted code, or consumed GPU memory in a way that crossed a budget, the final function return is not enough. The buyer needs to reconstruct the serverless-runtime event that produced the result in the first place.

The joinability gap behind most serverless-AI incident reviews

Modal Functions, the customer's application, the Modal Sandbox runtime, the GPU memory snapshot layer, the Volume / Secret / Dict state primitives, and the downstream Webhook / Schedule / Queue handler can each produce useful telemetry. The audit problem is that those records often use different correlation identifiers and retention windows. A Modal function log may show a container_id and image_digest while the application trace shows only a final return value. The downstream Webhook handler log may record a downstream call without preserving the sandbox-exec-evidence that justified it. A defensible trail needs one tenant-scoped correlation ID across every layer.

The eight rows that should join to one serverless-AI decision

  1. Function row. Tenant, workspace_id, project_id, user or service principal, timestamp, application release, function_id, app_id, and the original function call or canonical hash.
  2. Image row. Image identifier, revision or hash, base-images, modal.com runtime version, dependencies lockfile, and any modal.Image.build() invocation signature.
  3. Scheduler row. Modal project, regional endpoint, container-pool id, cold-start-vs-warm-start attribution, init-script-execution, and image-pull-time-capture.
  4. Sandbox row. Sandbox id, container_id, image_digest, command, env, mounted secrets, started_at timestamp, and process-tree capture at sandbox launch.
  5. GPU snapshot row. GPU model, CUDA state (allocated / reserved / free), tensor state (dtype / shape / device), allocator state (caching allocator + fragmentation), captured at function start, end, OOM, and SIGTERM.
  6. Volume / Secret / Dict row. Volume id, mount_path, read_only flag, mounted_at timestamp, fs_state_hash; secret_id, rotated_at, version_used; dict_id, mutation_event_id, value_hash.
  7. Webhook / Schedule / Queue row. Webhook ingress source_ip, payload_hash, handler_function, invoked_at; Schedule cron + last_trigger; Queue event id + handler + delivery-attempts.
  8. Action row. Human-approval state, downstream tool name, parameters hash, object or transaction ID, result, rollback event, and incident reference.

How this maps to SOC 2 and the EU AI Act

For a security reviewer, each of those rows has to map to a control. The Function row + Image row + Scheduler row establish SOC 2 CC7.2 (system operations) and EU AI Act Article 12 (record-keeping) by recording what code ran, on what image, and on what container-pool. The Sandbox row establishes SOC 2 CC6.1 (logical access) and EU AI Act Article 14 (human oversight) for untrusted-code-execution surfaces. The GPU snapshot row establishes SOC 2 CC7.3 (monitoring of outputs) and EU AI Act Article 15 (robustness + accuracy) by capturing CUDA + tensor + allocator state at decision points. The Volume / Secret / Dict row establishes SOC 2 CC6.1 + GDPR Article 30 (records of processing activities) for state-evidence. The Webhook / Schedule / Queue row establishes SOC 2 CC6.6 (network controls) + EU AI Act Article 12 for ingress-evidence. The Action row ties all of the above to a downstream business action so that an incident review can be reconstructed end-to-end.
