Five buyer-facing joins

  1. Unified-search answer provenance across connected sources: per-answer citation set with the Google Drive file IDs + Slack channel/message IDs + GitHub commit SHAs + Asana task IDs + Airtable record IDs that produced the answer; per-tenant ACL enforcement at each connector; LLM sub-processor invoked if LLM-augmented re-ranking is enabled; per-source retention window; per-connector region routing; exportable as a single artifact for procurement review.
  2. Verified topic-hierarchy + freshness state per KB article: per-article verification identity + timestamp + role; what changed since verification (diff hash); downstream search-result cache invalidations; verification-badge TTL (90 days? 180 days? configurable?); search-answer behavior when pointing at unverified topics (suppressed / flagged / down-ranked); persistence in Postgres + cache + search index.
  3. Permissions cascade across integrations: Google Workspace OAuth scopes per Drive file (least-privilege vs. broad tenant-wide); Slack channel ACL propagation including private-channel content; GitHub repo + branch ACL including private-repo code visibility; Asana workspace + project ACL; Airtable base + table ACL; Okta group-membership-driven role assignment. The unified-search result must respect every ACL on every connector simultaneously — and the test surface must prove a user who lacks Drive access to file X cannot see file X's content even if another connector returned a reference to file X.
  4. SOC 2 Type 2 + GDPR + ISO 27001 evidence packet per-tenant: annual SOC 2 Type 2 report excerpt scoped to the customer's tenant; GDPR Art. 28 sub-processor list scoped to the tenant's connectors (which of Stripe + Google Cloud Platform + Google Workspace OAuth + Slack + GitHub + Asana + Airtable + Okta + OpenAI/Anthropic are active for THIS tenant); ISO 27001 Statement of Applicability mapping to the customer's data classification; EU AI Act Art. 14 human-oversight operational record if the customer's tenant uses LLM-augmented unified-search.
  5. Sub-processor cascade DPA + 14-day change-notification SLA: Stripe (billing) + Google Cloud Platform (hosting) + Google Workspace OAuth (Drive) + Slack (channels) + GitHub (repos) + Asana (tasks) + Airtable (bases) + Okta (identity) + OpenAI/Anthropic (if LLM-augmented unified-search is enabled) — 8-12 sub-processors per answer for a customer using the full integration catalog. Flow-down DPA per GDPR Art. 28(2) + 14-day change-notification SLA; published list of LLM sub-processors used for unified-search re-ranking; customer-facing change-notification mechanism for connector additions (e.g. when Slab adds Notion as a connector, existing customers notified within 14 days).

Common evidence failures

  • Showing a unified-search answer without the per-connector citation set (Drive file IDs + Slack message IDs + GitHub commit SHAs + Asana task IDs + Airtable record IDs), the per-connector ACL snapshot, the LLM sub-processor invoked, and the retention window.
  • Logging a verified-article change without the diff hash, the downstream search-result cache invalidations, the verification TTL, and the behavior for search answers pointing at unverified topics.
  • Returning a unified-search result that crossed per-connector ACL boundaries (e.g. a user lacking Drive access to file X seeing file X's title in the search snippet because another connector returned a reference to file X).
  • Promoting an LLM-augmented unified-search ranking without the EU AI Act Art. 14 human-oversight operational record (who can override the ranking, what audit trail captures the override, what retention window applies).
  • Deleting a Slab tenant while omitting per-connector source copies, unified-search indexes, verified-article freshness state, LLM sub-processor invocation logs, evaluations, replicas, backups, and Stripe billing records — or omitting a connector addition from the 14-day change-notification SLA cascade.
