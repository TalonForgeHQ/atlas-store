Why this matters in 2026

Procurement, legal, and security teams now ask AI agent vendors for evidence that goes beyond a static SOC 2 Type II report. The 2026 diligence runbook expects four artifacts in the same packet:

  1. The system description that names the trust service criteria (Security, Availability, Processing Integrity, Confidentiality, Privacy) covered, the service commitments, and the supporting infrastructure.
  2. The control matrix with control identifiers, owners, frequencies, sample sizes, and evidence-of-evidence links.
  3. The exception register with the deviation, root cause, remediation, and residual risk for each failed test of operating effectiveness.
  4. The AI-specific receipt that joins a per-tenant id, per-workspace id, per-request id, model + model-version hash, prompt hash, completion hash, tool-call id, MCP server id, fallback chain id, guardrail decision id, human override id, token spend in USD, latency in ms, and audit-export id.

TSC control coverage an AI agent vendor must demonstrate

  1. CC1 — Control Environment. Code of conduct, background checks, security awareness training, role-based access provisioning with quarterly attestations.
  2. CC2 — Communication and Information. Service descriptions, sub-processor list, status page, customer-advisory cadence, and AI-specific incident comms templates.
  3. CC3 — Risk Assessment. Annual risk register, AI-risk taxonomy (prompt-injection, jailbreak, sensitive-log leakage, model exfiltration, MCP-server compromise), and per-model residual-risk rating.
  4. CC4 — Monitoring Activities. Continuous control monitoring for prompt-hash deviation, completion-hash anomaly, fallback-chain overuse, guardrail-block rate, token-spend ceiling, and human-override ratio.
  5. CC5 — Control Activities. Dual-control for model-version changes, MFA on production, immutable audit log, signed audit-export-id, and per-tenant key encryption.
  6. CC6 — Logical and Physical Access. Tenant isolation, per-virtual-key rate-limit, least-privilege IAM, quarterly access review, secrets in HSM/KMS, physical SOC 2 attestation for HQ + colos.
  7. CC7 — System Operations. Backup + restore drills, incident runbook, AI-agent postmortem, observability, and rollback rehearsal.
  8. CC8 — Change Management. Model version control, prompt-template version control, MCP-server version control, feature-flag audit, and per-change approver trail.
  9. CC9 — Risk Mitigation. Insurance, business continuity, vendor sub-processor risk reviews, and BCP/DR evidence with annual tabletop.

AI-specific columns the receipt must include

The receipt is the per-call join that auditors will sample. The minimum column set Atlas uses for AI-agent-vendor evidence-gap maps is:

tenant_id + workspace_id + request_id + session_id + user_id + model + model_version_hash + prompt_hash + completion_hash + tool_call_id + mcp_server_id + mcp_tool_name + guardrail_decision_id + fallback_chain_id + semantic_cache_hit_flag + retry_count + rate_limit_policy + input_tokens + output_tokens + cost_usd + latency_ms + status_code + human_override_id + audit_export_id

Failure modes the audit will catch

  1. Prompt-hash reused across tenants — indicates cross-tenant cache bleed.
  2. Completion-hash recorded without corresponding prompt-hash — incomplete chain of custody.
  3. Tool-call-id missing for an MCP-enabled call — no attribution for downstream side effects.
  4. Fallback-chain-id recorded but no fallback decision timestamp — cannot trace the failover path.
  5. Guardrail decision recorded without rule-version — cannot prove which rule fired.
  6. Token cost in USD missing for a sampled call — no FinOps reconcilability.
  7. Human override id missing for a sampled high-risk call — no accountability trail.
  8. Audit-export-id not deterministic — cannot re-run the export and reproduce the sample.

Sample-of-one question set for the diligence interview

  1. Which TSC criteria are in scope, and which trust-service criteria are explicitly out of scope?
  2. Which audit window does the latest Type II report cover, and when does the next one drop?
  3. Which sub-processors had a material change in the window, and how was the customer notified?
  4. Which exceptions failed in the window, what was the remediation, and what is the residual risk?
  5. Can the vendor reproduce the audit_export_id for a sampled call on demand?
  6. What is the per-tenant encryption boundary, and is the key per-tenant or per-vendor?
  7. What is the breach-notification SLA, and how does the vendor measure it?
