Five buyer-facing joins the Together AI audit scores

  1. Request → endpoint → model receipt. Can the buyer export one record that joins request ID, tenant, endpoint type, model ID and version, container or deployment revision, region, start/end time, token count, and cost? Serverless, provisioned-throughput, dedicated-model, and dedicated-container routes should remain distinguishable after export.
  2. Endpoint → GPU isolation receipt. For a dedicated deployment, can Together AI show which GPU family, cluster, tenant boundary, network policy, autoscaling event, and failover event handled the request without exposing another customer's identifiers? The evidence request covers GB300, GB200, B200, H200, and H100 without assuming that every family is available in every region or product tier.
  3. Model → storage → fine-tune lineage. Can the buyer join base-model version, training dataset hash, managed-storage object version, fine-tune configuration, resulting artifact, access policy, and retention window? This is the boundary between an inference receipt and a reproducible model-shaping receipt.
  4. Evaluation → release → human decision. Can an evaluation result be tied to the exact candidate model, benchmark version, threshold, reviewer, accept/reject decision, release revision, and rollback event? The checklist asks for evidence of the decision; it does not assume a specific Together AI approval workflow exists.
  5. Export → deletion proof. Can a customer export the joined record and then prove deletion across prompts, outputs, logs, caches, stored datasets, fine-tuned artifacts, evaluation data, and backups? The expected receipt names each scope, completion time, exception, and sub-processor dependency.
