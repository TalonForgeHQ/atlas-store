Five buyer-facing joins the Fireworks 756 audit scores

  1. Request → route → model → region → LoRA-adapter join. What does the Fireworks public surface publish for joining a single inference call to its full lineage, including the active LoRA-adapter identity? Cursor/Notion/Cresta security reviews need this join to answer "which model version served this answer, and which adapter wrote the tool call".
  2. Prompt-cache hit/miss evidence per request. When a Cursor Composer 2 Fast Apply call hits a Fireworks speculative-decoding cache, what is the buyer-facing evidence the cache returned a hit, what was the cache key, and what was the prompt prefix reused? This is the difference between "fast apply worked" and "fast apply is auditable".
  3. Tool-call boundary replay log. When a Gum Loop agent calls Fireworks for GLM-5.2 and the model emits a function-call, what is the buyer-facing log of the tool-call boundary (model emit → function name → argument schema → execution receipt → model re-entry)? This is the audit join an Agent Builder security review will fail without.
  4. Cost-token receipt per agent step. Cresta reports 100× cost reduction vs. GPT-4 via Multi-LoRA. The audit asks Fireworks to publish the per-step cost-token receipt (input tokens + output tokens + cached tokens + speculative-decoding tokens + LoRA-adapter overhead) so a buyer can reconcile the cost claim against the actual log.
  5. Prompt-injection firewall log + export/deletion receipt. OWASP LLM01 prompt-injection is the dominant threat model for production inference. The audit asks Fireworks to publish the firewall log shape (input-pattern, model-output-pattern, jailbreak-pattern) and the export/deletion receipt (tenant → model call → retained log → export → deletion).
