2. The 60+ column per-trace-id → per-observation-id → per-span-id → per-generation-id → per-score-id → per-event-id join-table

The canonical audit surface for any LLM-observability platform is the per-trace-id lineage that survives end-to-end across one user message → one agent run → N tool calls → M LLM calls → K retrieval calls → J judge calls → one verdict → one drift score → one billing event. For Langfuse specifically, this lineage is exposed at the column level in the ClickHouse trace-storage schema with the following surface:

  • per-trace-id: the canonical root identifier for one logical user request that may span multiple observations + generations + spans + scores + events
  • per-observation-id: the canonical child identifier for one observation (which may be a generation / span / event / score, depending on the observation-type field)
  • per-span-id: the canonical child identifier for one span-type observation (e.g., a tool call, retrieval call, or arbitrary work-unit that has a start + end time + parent-id)
  • per-generation-id: the canonical child identifier for one LLM-call-type observation (with per-model-id + per-prompt-id + per-prompt-version-id + per-completion-id + per-token-usage-input + per-token-usage-output + per-cost-usd lineage)
  • per-score-id: the canonical child identifier for one eval-score (with per-evaluator-id + per-evaluator-version-id + per-eval-run-id + per-eval-dataset-id + per-eval-dataset-version-id + per-judge-model-id + per-judge-prompt-template-id + per-ground-truth-id lineage)
  • per-event-id: the canonical child identifier for one event-type observation (with per-event-type + per-event-data-json + per-event-source-id lineage)
  • per-session-id: the canonical thread-identifier that groups multiple trace-ids into one user-conversation
  • per-user-id: the canonical end-user identifier (set via Langfuse SDK's update_current_observation(user_id=...) or update_current_trace(user_id=...))
  • per-tenant-id: the canonical multi-tenant isolation identifier (Langfuse Cloud uses per-project-id + per-org-id; self-hosted OSS uses per-project-id without per-org-id)
  • per-project-id: the canonical project-isolation identifier (every Langfuse deployment has at least one project, and per-project-id is the canonical isolation boundary for traces + observations + scores + datasets)
  • per-API-key-id: the canonical credential-isolation identifier (Langfuse Cloud API keys + OSS service-account keys all have a per-key-id; the trace-schema should record which key ingested each observation)
  • per-deployment-region: the canonical region-isolation identifier (Langfuse Cloud US vs EU cluster selection; OSS deployments are region-bound by the hosting geography)
  • per-billing-event-id: the canonical cost-attribution identifier (per-trace-id-cost-usd + per-observation-id-cost-usd + per-generation-id-cost-usd for Cloud paid tiers)

Audit verdict (column-level join-table): the per-trace-id lineage is the canonical audit-trail surface. The OSS-First nature means the schema is inspectable in the ClickHouse tables directly (Langfuse OSS exposes the ClickHouse schema in the public GitHub repo), which is a meaningful SOC 2 CC7.2 advantage over Cloud-only vendors where the schema is opaque.

3. The OSS-to-Cloud handoff trace-propagation audit gap

The canonical audit gap unique to Langfuse (and not present in Arize Cloud-only managed-service or Helicone Cloud-only proxy) is the OSS-to-Cloud handoff trace-propagation surface. Most engineering teams we audit run Langfuse OSS in dev (for cost + per-tenant control + data-residency) and Langfuse Cloud in prod (for managed-uptime + ClickHouse-ops-free + per-region failover). When teams migrate traces between environments — or run hybrid OSS-dev + Cloud-prod simultaneously — the audit question becomes: does the per-trace-id → per-observation-id → per-span-id → per-generation-id → per-score-id → per-event-id → per-session-id → per-user-id → per-tenant-id → per-project-id → per-API-key-id → per-deployment-region → per-billing-event-id lineage survive the environment boundary, or does Cloud re-mint identifiers on import?

Five sub-questions for the OSS-to-Cloud handoff:

  1. Does the per-trace-id survive migration? Specifically, if trace T_001 was written to OSS-DEV and is later imported to Cloud-PROD, does Cloud store it as T_001 (preserved) or as T_new_001 (re-minted)?
  2. Does the per-observation-id + per-span-id + per-generation-id + per-score-id + per-event-id lineage survive, or does Cloud re-mint all child identifiers while only preserving the per-trace-id?
  3. Does the per-session-id + per-user-id + per-tenant-id + per-project-id + per-API-key-id migration preserve the source-environment identifiers, or does Cloud require re-mapping to a new per-project-id + per-API-key-id?
  4. Does the per-prompt-id + per-prompt-version-id history follow the trace (so that audit reconstruction shows which prompt-version was active at observation time), or is the prompt-history only available in the source environment?
  5. Does the per-deployment-region + per-billing-event-id lineage carry over, or does the Cloud environment re-compute the cost-attribution + region-isolation on import?

Audit verdict: the OSS-to-Cloud handoff trace-propagation surface is the canonical SOC 2 CC7.2 + ISO 42001 9.4 concern for any LLM-observability platform that ships both OSS + Cloud surfaces. Langfuse is the ONLY Tier-1 vendor in the llm_observability cohort with this hybrid-deployment surface (Arize is Cloud-only + Phoenix OSS as separate ingest point; Helicone is Cloud-only proxy; Honeycomb is general-purpose observability without LLM-specific primitives).

6. ClickHouse trace-storage encryption + per-tenant-CMK/BYOK + per-region-KMS-key-id

The canonical audit surface for any ClickHouse-powered observability platform is the encryption-in-rest + per-tenant-CMK/BYOK + per-region-KMS-key-id model. Langfuse Cloud provides per-tenant-CMK/BYOK on Enterprise tier (BYOK via AWS KMS / GCP KMS / Azure Key Vault) and per-region-KMS-key-id (the KMS key for the US region is separate from the KMS key for the EU region). For self-hosted OSS deployments, the per-tenant-CMK/BYOK + per-region-KMS-key-id are the responsibility of the deploying entity (ClickHouse supports per-table encryption with per-key-id, which the deployer can configure for per-tenant-isolation).

Audit verdict: per-tenant-CMK/BYOK + per-region-KMS-key-id are the canonical SOC 2 CC6.1 + GDPR Art. 28 + EU AI Act Art. 10 + ISO 27001 A.8.24 (use of cryptography) surface. Langfuse Cloud Enterprise provides per-tenant-CMK/BYOK + per-region-KMS-key-id out of the box; self-hosted OSS deployments can configure equivalent primitives via ClickHouse's per-table encryption + per-key-id.

7. Prompt-management versioning system audit

Langfuse's prompt-management system maintains git-style prompt-version history with per-prompt-id → per-prompt-version-id → per-prompt-deployment-id → per-rollback-id → per-A-B-test-id → per-feature-flag-id lineage. The audit question is whether the prompt-version history is joined to the trace-table at the per-observation-id level (so that an auditor reconstructing one trace can see which prompt-version was active at the time of the observation) or whether the prompt-history is a separate table that requires manual joining.

Audit verdict: Langfuse's prompt-versioning system is the canonical LLM-prompt-management audit-trail surface, and the per-observation-id → per-prompt-version-id join is available at the column level (the prompt-version-id is recorded on each generation-type observation).

8. Evaluation + scores layer audit

Langfuse's evaluation + scores layer maintains per-score-id → per-evaluator-id → per-evaluator-version-id → per-eval-run-id → per-eval-dataset-id → per-eval-dataset-version-id → per-judge-model-id → per-judge-prompt-template-id → per-ground-truth-id lineage. The audit question is whether the score-history is joined to the trace-table at the per-trace-id + per-observation-id level (so that an auditor can see which eval-score was generated for which observation at which point in time) or whether the score-history is a separate table that requires manual joining.

Audit verdict: Langfuse's evaluation layer is the canonical LLM-eval audit-trail surface, and the per-trace-id → per-score-id + per-observation-id → per-score-id joins are available at the column level (each score is associated with a trace-id and optionally an observation-id).

9. Cross-tenant + cross-project + cross-API-key isolation-evidence

For SOC 2 CC6.1 + GDPR Art. 28 + CCPA/CPRA + EU AI Act Art. 10 + ISO 27001 A.8.22 (segregation in networks) compliance, the audit must demonstrate per-tenant-id + per-project-id + per-API-key-id + per-workspace-id isolation-evidence. Langfuse Cloud provides per-project-id as the canonical isolation boundary (every trace + observation + score + dataset + prompt is scoped to a single project-id), with per-org-id as the parent grouping for multi-project customers. For self-hosted OSS deployments, the isolation is at the per-deployment level (single-tenant OSS deployments) or at the per-project-id level (multi-tenant OSS deployments with the enterprise feature flag enabled).

Audit verdict: per-project-id + per-org-id + per-API-key-id isolation is the canonical SOC 2 CC6.1 + GDPR Art. 28 + ISO 27001 A.8.22 surface. Langfuse Cloud provides this out of the box; self-hosted OSS deployments can configure equivalent primitives.
