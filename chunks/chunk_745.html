Visier's first-party product surface positions Vee as the People Analytics AI Agent (trusted by over 2 million users), bundled with the Visier MCP Server that publishes governed people insights to any MCP-enabled external agent with a single connection. Vee is delivered natively to Microsoft Copilot, Microsoft Teams, Slack, intranet, and internal chatbots, and inherits the asker's role-based access from Visier's dynamic RBAC model. The Visier Real-Time People Data Platform reports 65,000 companies trusting Visier, $1.8M+ reduced cost of ownership, and 7.5 months to payback. That footprint means a static SOC 2 review is not enough; a buyer needs an event-level chain that joins each workforce question to the prompt/model version, source-system connector, data-model version, metric definition, role-filter, generated answer, and downstream manager action.

Gap 1 — Question-to-answer provenance (Vee + MCP)

For each Vee answer (in-product or MCP-published), preserve the question, account/tenant, product surface, prompt/model version, source-system connector, data-model version, metric definition, role-based access filter, generated output, provenance hash, and timestamp.

  • Question ID + asker role/region/tenant
  • Selected Vee prompt/model version + connector (Workday / SAP / Oracle / Salesforce / etc.)
  • Data-model version + metric definition
  • Role-based access filter result
  • Generated answer with provenance hash + timestamp

Gap 2 — Vee Boards personalization chain

Vee Boards surface AI-augmented insight boards to leaders. For each surfaced board, preserve the leader/role, board definition, anomaly/outlier detection rule, confidence score, surfaced insight, downstream manager action (drill-down, share, follow-up workflow), reviewer, and outcome.

  • Leader role + tenant/region
  • Board definition + anomaly rule version
  • Confidence score + surfaced insight
  • Downstream action + reviewer + outcome

Gap 3 — Workforce plan and scenario governance

For each workforce plan or scenario, preserve the assumption set, model version, sensitivity inputs, simulated outcome, approval gate, reviewer, published plan version, and rollback. Without these, a buyer cannot answer "why was this headcount plan approved?"

  • Assumption set + sensitivity inputs
  • Model version + simulated outcome
  • Approval gate + reviewer + timestamp
  • Published plan version + downstream consumers

Gap 4 — Source-system change lineage

When an HRIS (Workday, SAP SuccessFactors, Oracle HCM), ATS (Greenhouse, Lever, Workday Recruiting), payroll, or CRM source-system schema or field changes, the data-pipeline ingestion run, data-quality check, affected dashboards/boards/agents, customer-visible change log, and rollback path must be reconstructible.

  • Source-system change ID + ingestion run ID
  • Data-quality check + pass/fail
  • Affected dashboards/boards/agents list
  • Customer-visible change log + rollback path

Gap 5 — Access, correction, retention, deletion, and model/policy change evidence

For each access, correction, retention, deletion, or model/policy change request, identify every affected tenant, region, dashboard, board, agent conversation, MCP-published insight, and the reproducible export hash a customer can use as audit evidence.

  • Request type + tenant/region + timestamp
  • Affected dashboards/boards/agents/MCP insights
  • Reprocessed model/policy version + delta
  • Export hash + retention/deletion receipt
