Why multimodal lineage breaks

Vision, audio, text, and sensor projects rarely move through one system. Raw objects may remain in customer storage while annotation tasks, notebook revisions, MLflow runs, model versions, and deployment approvals live elsewhere. A dashboard can show each stage while still failing to prove that a deployed model came from an exact, approved source-data and label state.

DagsHub's repository, data-engine, annotation, experiment-tracking, model-management, pipeline, and deployment surfaces create the ingredients for a replayable chain. The control objective is to join those ingredients into one tenant-scoped receipt that survives UI changes and can be exported for buyer diligence.

Five controls for defensible replay

  1. Version the data and labels separately: a dataset revision must not silently inherit a changed annotation schema.
  2. Pin code to every run: preserve the repository or notebook commit beside the MLflow run identifier.
  3. Join approval to deployment: a model version should not reach a serving target without a resolvable promotion record.
  4. Record topology: identify whether evidence originated in Team cloud, customer storage, a VPC, OpenShift, or an air-gapped deployment.
  5. Export one replay bundle: serialize the full chain, validate tenant scope, and sign it with replay_hash.

Audit questions the map answers

  • Can a deployed model resolve to the exact source objects and dataset version?
  • Can each label resolve to an annotation task, schema version, and annotator?
  • Can an MLflow run resolve to the notebook or repository commit that executed?
  • Can an approval be distinguished from a deployment event?
  • Can the replay prove that no identifier crossed organization or tenant boundaries?
