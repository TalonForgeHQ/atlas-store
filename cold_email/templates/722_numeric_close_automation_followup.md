## Five-join buyer-ready evidence audit (procurement / Risk / IT / Compliance / Finance closes)

### Join 1 — Per-source-system version pinning across the close pipeline
Buyer question: "When an ERP / GL / sub-ledger / bank-API source changes, can Numeric pinpoint exactly which close-task, flux-analysis output, account-reconciliation packet, and journal entry was generated from which source-system version?"

Required evidence on Numeric's side: per-close-cycle + per-source-system + per-source-version + per-extraction-timestamp + per-reconciliation-packet-version + per-flux-analysis-version + per-journal-entry-version + per-close-task-owner audit-trail. The "GAAP close-cockpit" model requires every line in a close to be reproducible from a specific (source-system, source-version, extraction-time, Numeric-MCP-version, AI-flux-version) tuple. Auditors under PCAOB AS 2315 + AS 2401 must be able to trace every account balance back to source documents and the controls over them.

### Join 2 — Per-accountant-reviewer-version and per-approval-event cascade
Buyer question: "When an accountant edits a flux-analysis, restates an account reconciliation, approves a journal entry, or signs off a close task, can Numeric preserve who/when/version/role/region/device across the entire close?"

Required evidence: per-reviewer + per-reviewer-role + per-reviewer-region + per-reviewer-SSO-identity + per-reviewer-MFA-event + per-edited-flux-version + per-edited-reconciliation-version + per-approved-journal-entry-version + per-approval-event-timestamp + per-MCP-tool-call-version + per-Numeric-AI-recommendation-version cascade. This is the segregation-of-duties evidence that SOX 404 + SOC 2 CC6.1 + ISO 27001 A.9.4 require across the close.

### Join 3 — Per-Numeric-MCP / AI-flux / cash-matching version pinning
