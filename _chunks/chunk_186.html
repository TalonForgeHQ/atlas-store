Why W&B Deserves Its Own Audit-Trail Schema

W&B is the experiment-tracking anchor the rest of the cohort leans on. W&B Sweeps owns the per-run-id + per-run-version-id surface. W&B Artifacts owns the content-addressed per-artifact-id + per-artifact-version-id surface. W&B Prompts owns the per-prompt-id + per-prompt-template-version-id surface. W&B Traces + Weave own the per-trace-id + per-span-id surface. W&B Tables own the per-evaluation-id + per-ground-truth-id + per-human-review-id surface. Together they form the only experiment-tracking layer in the cohort where you can answer SOC 2 CC7.2, EU AI Act Art. 14, and HIPAA BAA with one SQL join — if you know which columns to ask for.

The schema below is the smallest set that survives all three audits simultaneously. Anything narrower loses either the per-trace-id provenance (required for SOC 2 CC7.2 + EU AI Act Art. 14) or the per-human-review-id link (required for EU AI Act Art. 14 + ISO 42001 §6.1.4 + HIPAA BAA).

The Canonical 22-Column W&B Join-Table

  1. per_run_id — logical experiment run (W&B Sweeps anchor)
  2. per_run_version_id — immutable hash of run config + git-sha at run time
  3. per_experiment_id — sweep / grid-search group
  4. per_artifact_id + per_artifact_version_id — content-addressed files (pickle, ONNX, SavedModel, GGUF)
  5. per_dataset_id + per_dataset_version_id — W&B Artifacts dataset version history
  6. per_model_id + per_model_version_id — W&B Models registry versions
  7. per_model_registry_version_id — W&B Model Registry alias + tag history
  8. per_model_deployment_id + per_deployment_version_id — W&B Launch rollout deployments
  9. per_prompt_id + per_prompt_template_version_id — W&B Prompts versions (the LLM-only column)
  10. per_completion_id — LLM output record with per-prompt-id link
  11. per_trace_id + per_span_id — W&B Traces + Weave distributed trace edge
  12. per_evaluation_id + per_evaluator_version_id — W&B Tables eval runs
  13. per_eval_run_id + per_ground_truth_id — ground-truth row-level evidence
  14. per_human_review_id + per_judge_model_id — reviewer + judge-model decisions
  15. per_tenant_id + per_workspace_id — multi-tenant + multi-workspace isolation
  16. per_billing_account_id + per_deployment_region — W&B Dedicated region + billing boundary

W&B column coverage (verified 2026-07-16 via wandb.ai/site/privacy):

  • W&B Sweeps contributes per_run_id + per_run_version_id + per_experiment_id (native).
  • W&B Artifacts contributes per_artifact_id + per_artifact_version_id + per_dataset_id + per_dataset_version_id (native, content-addressed).
  • W&B Models contributes per_model_id + per_model_version_id + per_model_registry_version_id (native).
  • W&B Prompts contributes per_prompt_id + per_prompt_template_version_id + per_completion_id (native — the only cohort sibling with first-class prompt versioning).
  • W&B Traces + Weave contributes per_trace_id + per_span_id (native, distributed).
  • W&B Tables contributes per_evaluation_id + per_eval_run_id + per_ground_truth_id + per_human_review_id + per_judge_model_id (native, the only cohort sibling with first-class ground-truth linking).

5 Audit Gaps — Mapped to Framework

  1. Reasoning-chain provenance — per-trace-id → per-span-id → per-prompt-template-version-id → per-run-version-id join. Required by SOC 2 CC7.2, EU AI Act Art. 14, NIST AI RMF MEASURE.
  2. Per-evaluation-id + per-ground-truth-id + per-human-review-id license-provenance. Required by EU AI Act Art. 53(d), ISO 42001 §6.1.4, FDA SaMD Class II. W&B is the only cohort sibling with first-class ground-truth versioning.
  3. Per-deployment-version-id + W&B Launch linkage. Required by SOC 2 CC7.3 + EU AI Act Art. 14. The gap: per-deployment-version-id lives outside W&B in most stacks.
  4. Cross-tenant + cross-deployment-region isolation — per-tenant-id + per-deployment-region + per-billing-account-id. Required by SOC 2 CC6.1, GDPR Art. 28, HIPAA BAA, FedRAMP Moderate, W&B Dedicated (single-tenant).
  5. WORM retention on Artifacts. 13-year content-addressed archive per GDPR Art. 17 deletion-propagation + EU AI Act Annex III + Aug 2026 GPAI enforcement.

14-Question Buyer Checklist (Run This Against Your W&B Stack)

  1. Can you trace one W&B Sweep run back to its exact per_run_version_id in under 60 seconds?
  2. Is the per_prompt_template_version_id recorded for every W&B Prompts call?
  3. Can you show the per_artifact_version_id linked to every per_run_id?
  4. Is the per_dataset_version_id immutable (content-addressed) in W&B Artifacts?
  5. Does the per_trace_id record carry the input prompts + completions (for replay)?
  6. Can you enumerate every per_tenant_id + per_workspace_id that ever touched a given run?
  7. Does your per_evaluation_id link to a per_ground_truth_id with timestamps?
  8. Is per_human_review_id linked to per_judge_model_id for the evaluator decisions?
  9. Can W&B Traces + Weave reconstruct a full per_span_id tree for a multi-agent run?
  10. Is the per_model_registry_version_id alias → version-id linkage tamper-evident?
  11. Do per-deployment-region copies exist for every W&B Launch deployment?
  12. Are per_run_version_id runs reproducible (deterministic seeds + pinned deps + git-sha)?
  13. Does your per_artifact_id carry the training-data summary (EU AI Act Art. 53(d))?
  14. Can you prove per_tenant_id + per_deployment_region isolation-evidence to a third-party assessor?

If you answered no to > 3 of these, the canonical 22-column W&B join-table is your remediation map.
