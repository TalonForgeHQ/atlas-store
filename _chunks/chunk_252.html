SOC 2 CC7.2 Evidence Trail — Per-DeepKeep-Firewall-API-call-id

The CC7.2 (System Operations — monitoring + incident response + evidence trail) requirement under SOC 2 Type II is satisfied when the service organization can produce a per-event audit row for every detection event with: timestamp + tenant-id + project-id + detection-id + score + action-taken (block/allow/redact/transform) + linked-LLM-call-id + linked-VLM-call-id + linked-agent-step-id + linked-billing-event-id.

DeepKeep's per-DeepKeep-Firewall-API-call-id lineage is the canonical anchor because every firewall detection event — whether against an LLM call, VLM call, or agent step — gets a single Firewall-API-call-id. Joining on that anchor produces:

EU AI Act Art. 12 (Logging) + Art. 15 (Cybersecurity)

EU AI Act Article 12 requires "automatic recording of events ('logs') over the lifecycle of the AI system" with sufficient detail to ensure traceability, auditability, and post-market monitoring. Article 15 requires "an appropriate level of accuracy, robustness, and cybersecurity" — and the Aug 2026 GPAI enforcement makes this mandatory for any vendor serving EU customers.

For DeepKeep — which serves enterprise customers with EU data residency requirements — the Art. 12 + Art. 15 binder is built from the same per-DeepKeep-Firewall-API-call-id lineage as the SOC 2 CC7.2 binder, but with two EU-specific additions:

  • per-AI-system-id — the AI Act treats the deployed AI system as a single auditable unit (under Annex IV technical documentation); DeepKeep needs to expose a per-AI-system-id grouping across all detection events touching that system.
  • per-EU-data-residency-id — EU customers require that all detection events + audit rows are stored in EU regions only; DeepKeep needs to expose a per-region-id flag on every audit row.

Article 15's accuracy + robustness + cybersecurity triad maps to DeepKeep's coverage: (a) accuracy = per-detection-score-id + per-evaluation-result-id (from AI Red Teaming); (b) robustness = per-AI-Red-Teaming-eval-id (from Vibe AI Red Teaming); (c) cybersecurity = per-ML-supply-chain-id + per-ML-trojan-id + per-ML-backdoor-id (from Model Scanning) + per-real-time-detection-id (from AI Firewall).

Israel Privacy Protection Law Dual-Stack

DeepKeep's Israel-headquarters creates a unique dual-stack requirement: EU AI Act + GDPR (for global customers) + Israel Privacy Protection Law (Protection of Privacy Law 5741-1981 + Amendment 13 + Amendment 14 + Privacy Protection Regulations Data Security 2017) (for Israeli data-subjects). The dual-stack is non-trivial because:

  • Israel's Privacy Protection Authority (PPA) Database Registration requirement means DeepKeep must register any database containing personal data with the PPA. Detection events that touch personal data (e.g. PII redaction events) require per-database-id registration evidence.
  • Amendment 14 (2024) introduced enhanced cybersecurity obligations for database holders — every database must have a written cybersecurity policy + a designated DPO + a breach-notification procedure (24-hour notification to PPA + affected data-subjects).
  • Privacy Protection Regulations Data Security 2017 specify the technical + organizational measures required — DeepKeep's SOC 2 Type II + ISO 27001 + ISO 9001 + GDPR certifications satisfy most of the 2017 requirements, but the per-detection-event audit row needs an additional per-Israeli-DPO-id + per-PPA-database-registration-id link.

The audit binder for Israeli customers adds 3 columns to the SOC 2 / EU AI Act binder: per-Israeli-DPO-id, per-PPA-database-registration-id, per-Israeli-data-subject-consent-id. DeepKeep's position as an Israeli-headquartered vendor is actually an advantage here — most US-headquartered competitors (Lakera Zurich, Protect AI Seattle, HiddenLayer Austin) don't have the dual-stack expertise.

VLM / Vision Coverage — Per-VLM-call-id Lineage

DeepKeep's DeepKeep for Vision (VLM) is the unique differentiator vs Lakera + Protect AI + HiddenLayer (which all focus on LLM-only or ML-supply-chain-only). The per-VLM-call-id lineage is:

  1. per-VLM-call-id — the top-level audit row for one VLM inference call
  2. per-image-id — the input image (or video frame)
  3. per-prompt-template-version-id — the text prompt that accompanies the image
  4. per-completion-id — the VLM-generated text output
  5. per-token-id — each output token (sub-token granularity for jailbreak-detection)
  6. per-vision-poisoning-id — if the input image contains adversarial perturbations that target downstream model behavior
  7. per-visual-prompt-injection-id — if the image contains text/visual elements designed to override the system prompt
  8. per-adversarial-perturbation-id — if the image contains pixel-level perturbations designed to flip the classification
  9. per-deepfake-detection-id — if the image is detected as synthetic / deepfake / GAN-generated

This 9-column lineage covers the full VLM attack surface. SOC 2 + EU AI Act auditors will look at this lineage specifically — Lakera + Protect AI don't have it.

AI Agent Scanner — Per-Agent-step-id Audit

The AI Agent Scanner is DeepKeep's runtime attack-surface scanner for agentic systems (LLM agents + tool-using agents + MCP-server-using agents + multi-agent systems). The per-Agent-step-id lineage is the canonical audit row:

  1. per-agent-id — the parent agent (which may span many steps)
  2. per-agent-step-id — one step in the agent's execution (one prompt + one tool-call + one response)
  3. per-tool-call-id — the tool call made at this step (or null if no tool was called)
  4. per-MCP-server-call-id — the MCP server call (or null if MCP wasn't used)
  5. per-RAG-retrieval-id — the RAG retrieval made at this step (or null if no RAG)
  6. per-agent-prompt-injection-id — if the step's prompt contains an injection attempt
  7. per-agent-step-poisoning-id — if the step's tool-output contains poisoning that targets subsequent steps
  8. per-DeepKeep-tenant-id — the tenant anchor (shared with LLM + VLM + Firewall lineages)

For agentic systems, this 8-column lineage is the per-step audit row. AI Agent Scanner's runtime mode means detection events fire in real-time as the agent executes, not just at request-time like a traditional API firewall.

Cross-DeepKeep-tenant Isolation Evidence

Cross-tenant isolation is the single hardest SOC 2 CC6.1 + GDPR Art. 28 + EU AI Act Art. 10 + Israel Privacy Protection Law requirement. DeepKeep needs to produce evidence that no detection event from Tenant A can be observed by Tenant B, and no detection event from Tenant A's Project X can be observed by Tenant A's Project Y.

The evidence binder for cross-tenant isolation has 5 columns:

  1. per-DeepKeep-tenant-id isolation-test-result — a quarterly pen-test report that exercises cross-tenant probe attempts and confirms zero leakage
  2. per-DeepKeep-project-id CMK/BYOK optionality — confirmation that customers can supply their own customer-managed keys (CMK) for project-level encryption-at-rest
  3. per-trace-no-leakage-evidence — a sampling-based test that 1% of all detection events are traced through the entire stack and confirmed to land only in the originating tenant's audit table
  4. per-tenant-residue-cleanup-procedure — documentation of the procedure for deleting all tenant data on contract termination (with completion-of-deletion timestamp)
  5. per-completion-of-deletion timestamp — the actual ISO-8601 timestamp when the tenant-residue cleanup completed

For Israeli customers, the per-tenant-residue-cleanup-procedure additionally requires PPA notification (per Israel Privacy Protection Regulations Data Security 2017, Section 11).
