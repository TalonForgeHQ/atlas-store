The 5 audit-trails itemized

1. Per-system risk-tier classification with documented trigger, version, reviewer, and date

EU AI Act Art. 6 + Art. 9 + Annex III require every AI system deployed in scope to be classified against the high-risk taxonomy with: what triggered the classification (use-case + sector + impact), what version of the classification rubric was applied (rubric-version + ISO 42001 control-mapping + Annex III cross-walk), who reviewed the decision (reviewer identity + role + date), and what the date-stamped re-evaluation cadence is. Drata's existing risk module supports per-control risk tiering but the AI-system-specific rubric mapping is custom per customer today.

2. Per-data-category provenance showing training / fine-tuning / RAG / tool-call / eval data sources + DPIA + retention

EU AI Act Art. 10 data-governance requirements map to: training-data source inventory with license + provenance + consent + DPIA attachment, fine-tuning-data source inventory with the same fields, RAG-data source inventory with retrieval-channel + freshness + redaction-policy, tool-call-data source inventory with tool-name + tool-vendor + tool-version + tool-data-residency, eval-data source inventory with eval-prompt-version + eval-judge-version + eval-human-reviewer-version. Drata's vendor-risk module covers (d) partially; (a), (b), (c), (e) are bespoke.

3. Per-model + per-prompt + per-tool-version manifest with change-history, eval-result attachment, and approval-record

EU AI Act Art. 11 technical-documentation requirements: per-AI-system model manifest (model-vendor + model-name + model-version + deployment-region + sub-processor-list + DPA-cascade + 14-day-change-notification SLA), per-prompt-version manifest (prompt-version + author + reviewer + approval-date + deployment-region + eval-result-attachment), per-tool-version manifest (tool-name + tool-vendor + tool-version + permissions + approval-record). Drata's control-library has a framework-mapping engine that can be extended to ingest prompt-version + tool-version manifests; the ingestion schema is the gap.

4. Per-incident + per-human-override log with severity classification + reviewer identity + response-time + resolution

EU AI Act Art. 12 logging + Art. 14 human-oversight requirements: per-incident log (incident-id + severity + AI-system-id + reviewer-identity + reviewer-role + detection-timestamp + response-timestamp + resolution-timestamp + postmortem-attachment), per-human-override log (override-id + AI-system-id + human-reviewer-identity + human-decision + AI-suggestion + confidence-score + override-reason + override-audit-trail). Drata's existing incident module supports manual entry well; the AI-system-specific override pattern (with confidence-score + AI-suggestion-attachment) is the gap.

5. Per-output + per-evaluation evidence pack showing accuracy / robustness / cybersecurity test methodology + metrics + thresholds + outliers + remediation

EU AI Act Art. 15 accuracy/robustness/cybersecurity requirements: per-output evaluation record (output-id + eval-prompt-version + eval-judge-version + eval-score + eval-threshold + eval-decision + reviewer-attachment), per-evaluation evidence pack (eval-suite-version + dataset-version + model-version-under-test + threat-model + red-team-results + cyber-test-results + outlier-cases + remediation-record). Drata's existing evidence-collection layer covers the framework-mapping half; the AI-output-specific eval-data ingestion is the gap.
