4. The data-downtime framing (Barr Moses's contribution)

Barr Moses coined data downtime as the data-equivalent of application downtime — the period when data is missing, incomplete, or wrong. Monte Carlo Data Observability treats that as a first-class SLO breach event (per-data-SLO-breach-event-id) with paged alert + on-call rotation. The downstream impact: every ML model that depends on that data asset has a per-ML-model-monitoring-event-id blast-radius row that auditors can reconstruct from the per-data-SLO-breach-event-id alone. This is the audit-trail pattern Cleanlab doesn't have — Cleanlab detects label errors after they've been written; Monte Carlo detects the data pipeline failure before downstream models consume bad data. Two complementary surfaces in the same ai_data_quality_observability cohort.

5. Vendor-DD vendor-of-record pattern

For vendor-DD under SOC 2 + ISO 27001 + EU AI Act Annex IV, the canonical walkthrough on a Monte Carlo instance is:

  1. Ask for the per-data-asset-id column on every anomaly event (Monte Carlo Data Anomaly Detection) — confirms the asset is queryable.
  2. Ask for the per-data-lineage-edge-id graph (Monte Carlo Data Lineage) — confirms the upstream / downstream chain is reconstructable.
  3. Ask for the per-data-SLO-breach-event-id + per-data-anomaly-detection-event-id join — confirms the on-call rotation is auditable.
  4. Ask for the per-ML-model-monitoring-event-id join to per-data-asset-id — confirms ML blast-radius is queryable for SOC 2 CC7.2 + EU AI Act Art. 12.
  5. Ask for the per-cross-tenant-isolation-flag evidence on a multi-tenant instance — confirms CC6.1 + GDPR Art. 28 cross-tenant isolation.
  6. Ask for the EU AI Act Art. 53(d) license-provenance table — flag a gap if Monte Carlo returns "we don't track per-training-corpus-id, only per-data-asset-id". This is the gap Cleanlab covers.
  7. Ask for the WORM retention + 6yr reconstruction drill — confirms 6yr evidence preservation per SOC 2 + EU AI Act Annex IV + ISO 42001 9.4.
