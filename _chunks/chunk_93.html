Why the catalog layer is the new audit-trail bottleneck

EU AI Act enforcement hits August 2, 2026 with full GPAI obligations, and the cross-catalog RBAC layer — Apache Gravitino (incubating metalake), Apache Nessie (Git-like catalog), Iceberg REST Catalog (Polaris + Tabular + Dremio + generic Iceberg REST), and Unity Catalog (Databricks + open-sourced OSS Unity) — is now procurement-critical evidence in every Fortune 500 SOC 2 Type II + ISO 42001 + EU AI Act Art. 10 audit. The reason: when a Fortune 500 deploys a Llama-3 RAG-pipeline + AI-agent-reasoning-chain workload that touches 3+ catalogs (one per engine: Snowflake Iceberg + Databricks Unity + AWS Glue + on-prem Hive), the auditor now asks "where is the cross-catalog grant chain reconstructed, and is the credential-vending flow auditable end-to-end across the catalog surface?" That question lands directly on the catalog-of-catalogs governance stack.

Apache Gravitino is the metalake (catalog-of-catalogs) under Databricks + Apache Foundation governance; it normalizes Hive Metastore + Iceberg REST + Glue + Unity Catalog + JDBC catalogs behind a single RBAC surface with tag-based access control + column-mask policies + row-filter policies. Apache Nessie is the Git-like catalog from Comcast + Apple + LinkedIn + Adobe + Dremio + Cloudera + Netflix; it versions lakehouse commits as Git-style branches + tags + hashes + merge-rebase operations. Iceberg REST Catalog is the standardized REST interface (Polaris + Tabular + Dremio Sonar + Lakekeeper + Apache Gravitino-issued Iceberg-REST + Snowflake Horizon Catalog + BigLake Iceberg Catalog) — every AI-pipeline that reads Iceberg tables now goes through one of these REST endpoints. Unity Catalog is the Databricks-launched open-source unified governance layer (OSS since 2024) that bundles tables + AI-models + files + dashboards + notebooks behind one RBAC surface.

These four open-source projects form the cross-catalog governance substrate of the modern multi-engine AI-data stack. When an AI agent queries Snowflake Iceberg tables, the query routes through an Iceberg REST catalog (Polaris or Tabular). When an auditor inspects the RAG pipeline, the auditor inspects the cross-catalog grant chain (Unity → Iceberg REST → Gravitino → Nessie) — not just the engine. When a regulator inspects the prompt-training-data-set, the regulator inspects the Gravitino tag-policy + Unity Catalog external-location-policy + Nessie branch-policy + Iceberg-REST OAuth2 token-scope — not just the row count.

The 16-column cross-catalog grant-chain reconstruction join-table

SOC 2 CC6.1 + EU AI Act Art. 10 + ISO 42001 6.1.4 + GDPR Art. 28 require an end-to-end join-table per cross-catalog grant that touches an AI-pipeline. Reference schema:

CREATE TABLE cross_catalog_grant_provenance (
  grant_event_id            UUID PRIMARY KEY,
  tenant_id                 UUID NOT NULL,
  user_id                   UUID NOT NULL,
  catalog_kind              TEXT,  -- GRAVITINO, NESSIE, ICEBERG_REST, UNITY
  catalog_endpoint          TEXT,  -- gravitino://, nessie://, iceberg-rest://, unity://
  principal                 TEXT,  -- OAuth2 client_id / SAML subject / Unity Service Principal
  role                      TEXT,  -- catalog-issued role
  privilege_chain           JSONB, -- chain of privileges: CATALOG_USE -> SCHEMA_USE -> TABLE_SELECT -> COLUMN_MASK_APPLY
  gravitino_tag_policies    JSONB, -- Gravitino tag-based access policies applied
  unity_external_locations  JSONB, -- Unity external-location policies
  nessie_branch             TEXT,  -- Nessie branch @ main / @ experiment-x
  nessie_commit_hash        TEXT,  -- Nessie Git-hash
  iceberg_rest_oauth2_scope TEXT,  -- OAuth2 token-scope issued
  iceberg_rest_credential_vended JSONB, -- short-lived S3/GCS/ABFS credential issued
  unity_model_grants        JSONB, -- Unity model-governance grants (for AI-model reads)
  unity_notebook_grants     JSONB, -- Unity notebook + dashboard grants
  downstream_llm_call_ids   UUID[],-- LLM calls that used the resolved dataset
  downstream_agent_decisions UUID[],-- agent decisions that used the resolved dataset
  audit_event_time          TIMESTAMPTZ NOT NULL,
  audit_event_hash          BYTEA  -- SHA-256 over all above fields for tamper-evidence
);
CREATE INDEX ON cross_catalog_grant_provenance (tenant_id, audit_event_time DESC);
CREATE INDEX ON cross_catalog_grant_provenance (catalog_kind, principal);

The join-table unifies grant events from all four catalogs into a single auditable surface. The auditor can reconstruct: "user X, on tenant Y, at time Z, resolved dataset D from catalog C, with privilege chain [a→b→c], via OAuth2 scope S, with short-lived credential K, which was used by N LLM calls and M agent decisions." That reconstruction is the SOC 2 CC6.1 evidence the auditor needs.

The 5 audit gaps every catalog-of-catalogs deployment must close

1. Credential-vending flow evidence (the highest-frequency auditor ask)

Iceberg REST Catalog (Polaris + Tabular + Dremio Sonar + Lakekeeper + Gravitino-Iceberg-REST) issues short-lived S3/GCS/ABFS credentials per query. The audit-trail surface is: who requested the credential, what scope was granted, how long was the TTL, what table was read, what agent downstream received the data. Most deployments log the issuance but not the downstream use — leaving a gap. Fix: every credential issued by Iceberg REST must carry a x-downstream-llm-call-id + x-downstream-agent-decision-id + x-ai-disclosure-label header per EU AI Act Art. 50, and the credential-issuance event must be joinable to the LLM-call event via the join-table above.

2. Gravitino tag-based access policy provenance

Apache Gravitino tags (PII, PHI, GDPR-special-category, EU-AI-Act-high-risk, AI-training-corpus) propagate across Hive + Iceberg REST + JDBC catalogs. The audit-trail surface is: which tags were applied to which tables, who applied them, what mask-policy + row-filter-policy resolved per query. Most deployments tag the tables but don't log the per-query mask-applied + row-filter-applied event. Fix: every Gravitino query-resolution event must emit a gravitino_tag_policies_resolved JSONB blob to the join-table.

3. Unity Catalog external-location + model-governance grants

Unity Catalog governs 5 surfaces: tables, AI-models, files, notebooks, dashboards. The audit-trail surface is: which models were read by which agent, what notebook lineage chain was traced, what external-location (S3/GCS/ABFS) credentials were issued. Most deployments grant model-access but don't log the per-model-call event with downstream-agent-decision linkage. Fix: every Unity Catalog model-read event must emit a unity_model_grants_resolved + unity_notebook_grants_resolved JSONB blob to the join-table per EU AI Act Art. 12 + Annex III 4.

4. Nessie branch + commit hash lineage

Apache Nessie versions lakehouse commits as Git-style branches. The audit-trail surface is: which branch was read, what commit hash, what was the merge-rebase chain. Most deployments log the branch but not the merge-rebase chain, leaving the auditor unable to reconstruct "was this AI-training-data-set read from the production branch or the experiment branch?" Fix: every Nessie read-event must emit nessie_branch + nessie_commit_hash + nessie_merge_chain JSONB blob to the join-table.

5. Cross-catalog WORM retention + tamper-evidence

SOC 2 CC7.2 + SEC 17a-4 + EU AI Act Art. 12 require WORM-locked retention of all grant-chain reconstruction evidence. The audit-trail surface is: who wrote the join-table rows, who could modify them, how is tamper-evidence enforced. Most deployments use a mutable log table. Fix: the join-table must be WORM-locked (S3 Object Lock + Glue Data Catalog lock + Databricks Delta-time-travel + Iceberg snapshot-isolation) with a per-row SHA-256 hash chained from row to row for tamper-evidence.
