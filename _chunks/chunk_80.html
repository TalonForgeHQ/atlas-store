3. Audit gap #1 — End-to-end ClickHouse + ClickPipes + ClickHouse AI functions provenance join-table

The audit-trail surface for ClickHouse AI Act Annex III 4 high-risk classification is the 16-column provenance join-table:

  1. clickhouse_insert_span_id — every row insert into ClickHouse generates an OpenTelemetry span with provenance metadata
  2. clickhouse_query_span_id — every SELECT query generates a span with prompt + response + vector-search-decision
  3. clickpipes_cdc_span_id — every Change-Data-Capture event from upstream Kafka / Postgres / MongoDB generates a span
  4. clickhouse_vector_search_decision_id — every cosine-similarity-search or HNSW index lookup generates a vector-search-decision row
  5. clickhouse_ai_function_call_span_id — every ClickHouse AI function call (text-classification + sentiment-analysis + embedding-generation + LLM-inference-via-Python-UDF) generates a span
  6. clickhouse_cloud_tenant_isolation_evidence_id — per-tenant isolation-evidence proof for cross-tenant no-leakage testing
  7. clickhouse_cloud_byok_evidence_id — per-tenant BYOK (Bring Your Own Key) configuration evidence
  8. clickhouse_cloud_kms_evidence_id — KMS key-rotation + access-policy evidence
  9. clickhouse_cloud_audit_log_event_id — ClickHouse Cloud's native audit-log-event row
  10. downstream_snowflake_state_change_id — every ClickHouse-to-Snowflake-CDC-export state-change event
  11. downstream_databricks_state_change_id — every ClickHouse-to-Databricks-Delta-Lake-export state-change event
  12. downstream_bigquery_state_change_id — every ClickHouse-to-BigQuery-export state-change event
  13. downstream_redshift_state_change_id — every ClickHouse-to-Redshift-COPY-export state-change event
  14. downstream_postgres_state_change_id — every ClickHouse-to-Postgres-FDW-export state-change event
  15. clickhouse_drift_detection_event_id — every drift-detector-decision row (data-drift + concept-drift + label-drift)
  16. clickhouse_guardrail_trigger_decision_id — every guardrail-trigger-decision row (PII-redaction + toxicity-filter + jailbreak-block)

This join-table is the single artifact the customer's CISO signs off on. It's the SOC 2 CC7.2 + ISO 27001 A.12.4 + ISO 42001 §9.4 + EU AI Act Art. 12 evidence base. The 16-column join-table is the only artifact that survives a 7-year WORM retention + quarterly reconstruction drill.

4. Audit gap #2 — ClickHouse AI functions training-corpus + fine-tune license-provenance (Art. 53(d))

EU AI Act Art. 53(d) requires GPAI providers to publish a training-data-summary sufficient to reproduce the model's training. ClickHouse's AI functions are derived from a combination of:

  • Pre-trained open-source LLMs (Hugging Face transformers + Anthropic + OpenAI + Mistral + Cohere) — each model has its own training-data-summary
  • ClickHouse-customer fine-tunes on per-tenant conversation-history + eval-history + drift-detection-history + guardrail-trigger-history
  • ClickHouse-Python-UDF call-spans that route to OpenAI / Anthropic / AWS Bedrock / Google Vertex AI / Mistral / Cohere / Together / Replicate / Hugging Face inference endpoints

The 9-column per-training-corpus-source join-table:

  1. clickhouse_ai_function_corpus_source_id — unique corpus-source-row identifier
  2. clickhouse_ai_function_corpus_source_type — pre-trained-model + customer-fine-tune + python-udf-call
  3. source_license_type — Apache-2.0 + MIT + BSL + commercial + per-customer-DPA
  4. source_copyright_holder — Anthropic + OpenAI + Mistral + Cohere + Hugging Face + customer-name
  5. source_attribution_chain — provenance chain back to original training-data-source
  6. source_copyleft_flag — boolean for copyleft-license propagation
  7. source_license_collision_flag — boolean for license-conflict detection (e.g., GPL + proprietary)
  8. downstream_model_derivative_use — what downstream-models use this corpus-source
  9. downstream_retention_until — 7-year WORM retention timestamp

    10. ClickHouse's unique position: ClickHouse is the open-source columnar OLAP database that stores the AI Act training-data-summary for downstream-model training runs. The corpus-source-rows live inside ClickHouse-as-audit-target, not outside it. This is fundamentally different from Snowflake's cloud-data-warehouse audit lane (which is closed-source) and Databricks' lakehouse-platform audit lane (which uses Delta Lake + Unity Catalog).

  10. per_customer_no_leakage_evidence — SHA-256 of the per-customer no-leakage test result
  11. per_tenant_data_source_no_leakage — boolean for downstream-data-source no-leakage
