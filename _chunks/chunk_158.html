3. The 4-layer reference architecture for AI trust layer audit-readiness

  1. Layer 1 — Inference Event Capture: every LLM call (input/output/tool-call/embedding) is captured with tenant_key, ai_system_id, inference_event_id, ts, and a stable per-event hash.
  2. Layer 2 — Trust Score + Drift Detection Engine: the AI trust layer computes a per-event trust score and a per-model drift delta from a baseline embedding-set; threshold-crossings flag drift_threshold_crossed = TRUE and emit a mitigation_action_event_id.
  3. Layer 3 — Prompt-Injection + Adversarial ML Classifier: every input prompt is classified for injection-adjacent patterns (jailbreak attempt, indirect prompt injection, model-DOS, system-prompt-extraction, training-data-extraction); high-confidence hits emit a prompt_injection_detection_event with the classifier model version.
  4. Layer 4 — Reviewer-in-the-Loop + AI-Bill-of-Rights-Evaluation: every drift_threshold_crossed = TRUE row triggers a reviewer-acknowledgement workflow; every AI-system-onboard event triggers an ai_bill_of_rights_evaluation_run that captures the 5-field evaluation reasoning-chain (input snapshot, result, confirmation timestamp, reviewer id, mitigation action event id).

4. The 14-question buyer checklist

  1. Is per-AI-system trust score captured per inference event with a stable trust_score_event_id joinable per tenant_key + ai_system_id?
  2. Is cross-model drift detection reason-code surfaced with per-baseline-embedding-id and per-mitigation-action-event-id joinable per model_id?
  3. Is prompt-injection detection classifier version captured per detection event so we can re-run detection against historical events if the classifier model is updated?
  4. Is per-inference-event prompt + output + tool-call retained for 90+ days with a tamper-evident hash chain?
  5. Is the reviewer-in-the-loop threshold documented in the trust layer policy and surfaced as a per-event reviewer_id + review_ts not-null constraint when threshold-crossed?
  6. Is ai_bill_of_rights_evaluation_run a first-class entity (not a free-text field) with the 5-field reasoning-chain captured per evaluation?
  7. Is per-tenant KMS-CMK-BYOK supported across the trust layer telemetry store (not just the inference API)?
  8. Is cross-tenant isolation enforced via cross_tenant_isolation_hash at the SQL query layer (no implicit cross-tenant aggregation)?
  9. Are trust layer events exportable to the auditor's preferred format (SQL / Parquet / CSV) within 24h of a written request?
  10. Is the AI trust layer wired into the existing AI Governance Platform (Credo AI / Holistic AI / DataRobot / SUPERWISE) or does it require a separate approval workflow?
  11. Is the trust score reasoning-chain reproducible from raw inference events without re-running the classifier?
  12. Is the prompt-injection classifier updated via a documented model lifecycle (version captured, prior-versions retained for replay)?
  13. Is there a per-reviewer acknowledgement SLA captured per drift_threshold_crossed = TRUE event?
  14. Is per-cross-border-transfer data residency supported (US / EU / in-customer-VPC) so that the trust layer itself does not become a GDPR Art. 44 / Art. 46 / SCCs violation?
