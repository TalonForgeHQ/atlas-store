3. The Five Canonical Plaid Audit-Trail Gaps

Across our vendor-DD audit-target work with Plaid customers (Rillet, Numeric, Ramp, Brex, Mercury, BILL, Modern Treasury, Tipalti, plus dozens of consumer-banking + investment + lending + crypto apps), we see the same five recurring gaps in the Plaid audit-trail surface:

Gap A — Per-row provenance is exposed via the same /processor/ endpoint, not a separate /audit/ endpoint. Most bank-aggregation vendors split the per-row-provenance into a separate product line (e.g. Plaid Ledger, Plaid Income Verification Premium, Plaid Identity Premium), which forces audit-target customers into a higher-tier contract just to walk-back from a downstream-AI-decision to the source-of-truth bank-feed-row. The right architectural choice is to expose the per-row provenance in the SAME API call as the production data (e.g. Transactions, Balance, Identity) — the audit-trail should not be a separate product, it should be a property of every per-row response.

Gap B — Plaid Link token rotation policy + OAuth state + Webhook delivery are not part of the per-Link-session provenance. For a Plaid customer running 500K+ end-users through Plaid Link, the auditor will want to walk-back from a per-Identity-verification-decision to the per-Link-session-OAuth-state, the per-Link-token-rotation-policy, the per-MFA-bit, the per-Webhook-delivery-id, and the per-credential-storage-encryption-key-id — in ONE query. Today these are scattered across the Plaid Link Token Webhook model + the Plaid Auth /auth/get + the Plaid Identity /identity/get endpoints.

Gap C — Cross-tenant isolation evidence is per-Plaid-Link-session, not per-customer-link-token. For multi-tenant Plaid customers (a SaaS that serves 10,000+ downstream tenants each with their own Plaid-Link integration), the per-tenant-Plaid-Link-session isolation needs to be provable — but Plaid's current isolation model is per-Plaid-Link-token, not per-tenant. The SOC 2 CC6.1 + GDPR Art. 28 + EU AI Act Art. 10 + FedRAMP Moderate audit-target surface requires per-tenant isolation, not per-token isolation.

Gap D — Per-decision confidence-score + human-review-required-bit are not first-class fields on the Plaid Signal / Plaid Identity / Plaid Liabilities response. For an AI-driven fraud-decisioning + credit-decisioning + KYC-decisioning pipeline that uses Plaid Signal + Plaid Identity + Plaid Liabilities as inputs, the downstream-AI-decision needs to record the per-input-decision-confidence-score + per-input-human-review-required-bit. Today these are absent from the Plaid response, which forces downstream-AI-vendors to infer confidence from the input data alone (a recipe for false-positive-rate explosion + downstream-AI-decision-poisoning risk per OWASP LLM Top 10 LLM01).

Gap E — WORM retention + cost-attribution join-table for the per-row audit-trail is not exposed via a separate endpoint. For SOC 2 CC7.2 + EU AI Act Art. 12 + SEC 17a-4 WORM + IRS 6001 + SOX 404 + GLBA + BSA-AML + FCRA + ECOA compliance, the per-Auth-credential-exchange + per-Transactions + per-Balance + per-Identity + per-Liabilities + per-Income + per-Signal + per-Transfer cost must be attributable to the per-customer + per-tenant + per-downstream-AI-decision. Today Plaid exposes per-API-call cost only via Plaid's Billing dashboard, not as a join-table field on the per-row response.
