Why a BrowserQL run needs more than video replay

Video proves that pixels moved. It does not prove which BrowserQL query or MCP instruction ran, which browser image and proxy region executed it, which tenant credential was available, why a target was selected, which policy approved the action, what external object changed, or whether the session was erased. A procurement-grade record must join intent, interpretation, authorization, action, result, and teardown.

The 18-field browser-infrastructure receipt

  1. session_id — immutable Browserless execution key
  2. query_or_instruction_version — BrowserQL, MCP, or API input hash
  3. browser_runtime_image — browser and automation build
  4. tenant_identity — customer and workspace isolation key
  5. credential_scope — least-privilege secret boundary
  6. proxy_region — egress tier and jurisdiction
  7. target_url — destination and origin chain
  8. pre_action_page_hash — DOM/screenshot state before action
  9. resolved_target — selector, coordinates, or semantic element
  10. action_input_hash — click, type, extract, or download input
  11. security_boundary_event — CAPTCHA, 2FA, login, or download gate
  12. policy_validation_result — allow, deny, retry, or escalate
  13. human_approval_override — approver identity and reason
  14. replay_artifact — screenshot, trace, or recording reference
  15. downstream_object_id — purchase, filing, ticket, or record changed
  16. post_action_page_hash — observed resulting page state
  17. resulting_state_hash — tamper-evident outcome record
  18. teardown_deletion_receipt — browser, cookies, tokens, and artifacts erased

BrowserQL decision replay

When a page changes, BrowserQL can preserve workflow intent while adapting execution. Auditability requires the exact query version, candidate page state, resolved target, retry branch, browser image, proxy route, and post-action state. The acceptance test replays a captured run against a controlled page revision and explains any different target or branch.

Cloud versus Self-Hosted responsibility parity

A regulated buyer needs one machine-readable matrix for Browserless Cloud and Self-Hosted. It assigns ownership for browser images, patch cadence, network egress, proxy routing, credentials, recordings, downloads, observability, retention, sub-processors, encryption keys, incident response, and deletion receipts. Moving a workflow on premises should change responsibility columns, not erase evidence fields.

Credential, CAPTCHA, and session boundaries

Every authenticated run should bind credential scope to one tenant, one session, one target origin, and one expiration window. CAPTCHA, 2FA, login, and unsafe-download events become structured boundary records. Session completion must revoke tokens, destroy temporary cookies, isolate downloaded files, and emit a teardown receipt that can be reconciled with infrastructure logs.

Five evidence gaps enterprise buyers will test

  1. Instruction-to-action traceability: BrowserQL or MCP input, resolved target, action, and result are joined in one receipt.
  2. Tenant and session isolation: cookies, recordings, downloads, and credentials cannot cross workspace or run boundaries.
  3. Deployment parity: Cloud and Self-Hosted export the same minimum receipt with explicit responsibility changes.
  4. Security-boundary integrity: CAPTCHA, 2FA, login, and download gates produce deterministic structured events.
  5. External mutation and teardown: irreversible actions carry authorization and downstream object ids, followed by provable cleanup.

FAQ

Can the receipt be exported? Yes. The deliverable defines stable JSON and CSV schemas so a SIEM, GRC platform, or customer audit binder can ingest each run.

Is a session recording sufficient? No. Recordings are replay artifacts; they must be linked to the instruction, browser build, credential scope, policy result, downstream object, and teardown receipt.

Does Self-Hosted remove processor risk? It reallocates responsibility. The matrix records which party owns infrastructure, browser patching, network egress, retention, keys, sub-processors, deletion, and incidents.

What is the acceptance test? One real BrowserQL or MCP workflow is captured, replayed after a controlled page change, compared across deployment modes, and verified through final session teardown.
